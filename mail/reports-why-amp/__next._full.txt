12:Tc7fc,<h2>为什么选择 Amp？</h2>
<blockquote>
<p>恭喜你安装了 Amp。本手册将帮助你充分利用它。</p>
</blockquote>
<p>Amp 是由 Sourcegraph 打造的、面向终端和编辑器的前沿编码智能体（Agent）。</p>
<ul>
<li><strong>多模型（Multi-Model）：</strong> Sonnet、GPT-5、快速模型——Amp 会根据每个模型的特长，综合使用它们。</li>
<li><strong>观点鲜明（Opinionated）：</strong> 你始终使用的是 Amp 最好的部分。如果我们不使用也不喜欢某个功能，我们就会砍掉它。</li>
<li><strong>处于前沿（On the Frontier）：</strong> 模型发展到哪里，Amp 就跟进到哪里。没有向后兼容，没有遗留功能。</li>
<li><strong>会话（Threads）：</strong> 你可以保存并分享你与 Amp 的交互。写代码时你不会不用版本控制，对吧？</li>
</ul>
<p>Amp 有 2 种模式：<code>smart</code>（无限制使用最先进的模型）和 <code>free</code>（免费，使用快速的基础模型）。</p>
<p><em>想要深入了解吗？请收听我们的 <a href="https://ampcode.com/podcast">Raising an Agent 播客</a>，其中记录了构建 Amp 最初几个月的过程，或者查看我们的 <a href="https://ampcode.com/fif">FIF</a>。</em></p>

<h2>使用 Amp</h2>
<h3>智能体模式 (Agent Modes)</h3>
<p>Amp 有 3 种模式：</p>
<ul>
<li><strong><code>smart</code></strong>：无限制地使用最先进的模型，以获得最大的能力和自主性。这是默认模式，使用 <a href="https://ampcode.com/#paid-usage">付费额度</a>。</li>
<li><strong><code>rush</code></strong>：更快、更便宜，但能力稍弱，适合小型、明确的任务。参见 <a href="https://ampcode.com/news/rush-mode">Rush 模式</a>。</li>
<li><strong><code>free</code></strong>：免费，使用快速的基础模型。参见 <a href="https://ampcode.com/#free">Amp Free</a>。</li>
</ul>
<p>在 CLI 中通过打开命令面板 (Ctrl+O) 输入 <code>mode</code> 来切换模式，或者在编辑器扩展的提示词输入框中选择模式。</p>
<h3>如何编写提示词 (How to Prompt)</h3>
<p>Amp 目前使用 Gemini 3 处理大多数任务，拥有高达 100 万 token 的上下文。为了获得最佳结果，请遵循以下准则：</p>
<ul>
<li>明确你的需求。与其问“你能做 X 吗？”，不如直接说“做 X”。</li>
<li>保持简短，保持专注。将非常大的任务分解为较小的子任务，每个会话（Thread）处理一个。不要要求智能体在一个修改了文档页面 CSS 的会话中去编写数据库迁移脚本。</li>
<li>不要让模型去猜。如果你知道如何实现你想要的效果——比如查看哪些文件、运行哪些命令——请将其放入你的提示词中。</li>
<li>如果你希望模型不编写任何代码，只进行研究和规划，请直说：“只规划如何实现这一点。<strong>不要</strong>编写任何代码。”</li>
<li>使用 <a href="https://ampcode.com/#AGENTS.md"><code>AGENTS.md</code> 文件</a> 来指导 Amp 如何运行你的测试和构建步骤，并避免常见错误。</li>
<li>如果会话积累了太多噪音，请放弃它。有时事情会出错，失败的尝试和错误信息会弄乱上下文窗口。在这些情况下，最好重新开启一个会话，使用干净的上下文窗口。</li>
<li>告诉智能体如何最好地检查其工作：运行什么命令或测试、打开什么 URL、读取哪些日志。反馈对智能体的帮助就像对我们的帮助一样大。</li>
</ul>
<p>会话中的第一个提示词分量很重。它为接下来的对话设定了方向。我们鼓励你认真对待它。这就是为什么我们在 Amp 中使用 Cmd/Ctrl+Enter 来提交消息——这是一个提醒，让你在提示词上多花点心思。</p>
<p>以下是我们使用 Amp 时的一些提示词示例：</p>
<ul>
<li>“让 <code>observeThreadGuidanceFiles</code> 返回 <code>Omit&#x3C;ResolvedGuidanceFile, 'content'>[]</code> 并从其返回值中移除该字段，同时更新测试。注意：省略它是为了节省视图 API 传输的数据量，因为这些地方不需要文件内容。” (<a href="https://ampcode.com/threads/T-9219191b-346b-418a-b521-7dc54fcf7f56">查看会话</a>)</li>
<li>“运行 <code>&#x3C;build command></code> 并修复所有错误”</li>
<li>“查看 <code>&#x3C;local development server url></code> 以检查这个 UI 组件。然后修改它，使其看起来更简约。通过截图该 URL 来频繁检查你的工作”</li>
<li>“对我打开的文件运行 git blame，找出是谁添加了那个新标题”</li>
<li>“将这 5 个文件转换为使用 Tailwind，每个文件使用一个子智能体 (subagent)”</li>
<li>“看一看 <code>git diff</code>——有人帮我构建了一个调试工具，可以直接用 JSON 编辑 Thread。请分析代码，看看它是如何工作的以及如何改进。[…]” (<a href="https://ampcode.com/threads/T-39dc399d-08cc-4b10-ab17-e6bac8badea7">查看会话</a>)</li>

<h3>AGENTS.md</h3>
<p>Amp 会在 <code>AGENTS.md</code> 文件中查找有关代码库结构、构建/测试命令和规范的指导。</p>
<table>
<thead>
<tr>
<th>文件</th>
<th>示例</th>
</tr>
</thead>
<tbody>
<tr>
<td><code>AGENTS.md</code> (位于当前工作目录、父目录及子树中)</td>
<td>架构、构建/测试命令、内部 API 概览、审查和发布步骤</td>
</tr>
<tr>
<td><code>$HOME/.config/amp/AGENTS.md</code> <br> <code>$HOME/.config/AGENTS.md</code></td>
<td>个人偏好、设备特定命令，以及在提交到仓库前你在本地测试的指导</td>
</tr>
</tbody>
</table>
<p>Amp 会自动包含 <code>AGENTS.md</code> 文件：</p>
<ul>
<li>当前工作目录（或编辑器工作区根目录）<em>以及</em>父目录（一直到 <code>$HOME</code>）中的 <code>AGENTS.md</code> 文件总是被包含。</li>
<li>当智能体读取子树中的文件时，该子树的 <code>AGENTS.md</code> 文件会被包含。</li>
<li><code>$HOME/.config/amp/AGENTS.md</code> 和 <code>$HOME/.config/AGENTS.md</code> 如果存在，都会被包含。</li>
</ul>
<p>如果在目录中不存在 <code>AGENTS.md</code>，但存在名为 <code>AGENT.md</code>（没有 <code>S</code>）或 <code>CLAUDE.md</code> 的文件，该文件将被包含。</p>
<p>在具有多个子项目的大型存储库中，我们建议保持顶层 <code>AGENTS.md</code> 的通用性，并为每个子项目的子树创建更具体的 <code>AGENTS.md</code> 文件。</p>
<p>要查看 Amp 正在使用的智能体文件，请运行 <code>/agent-files</code> (CLI) 或在发送会话的第一条消息后悬停在 X% of 968k 指示器上 (编辑器扩展)。</p>
<h4>编写 AGENTS.md 文件</h4>
<p>如果不存在 <code>AGENTS.md</code> 文件，Amp 会提议为你生成一个。你可以手动创建或更新任何 <code>AGENTS.md</code> 文件，也可以通过询问 Amp 来完成（<em>“根据我在这个会话中告诉你的内容更新 AGENTS.md”</em>）。</p>
<p>要包含其他文件作为上下文，请在智能体文件中 @提及它们。例如：</p>
<pre><code class="language-markdown">See @doc/style.md and @specs/**/*.md.

When making commits, see @doc/git-commit-instructions.md.
</code></pre>
<ul>
<li>相对路径是相对于包含提及的智能体文件进行解释的。</li>
<li>也支持绝对路径和 <code>@~/some/path</code>。</li>
<li>代码块中的 @提及会被忽略，以避免误报。</li>
<li>支持 Glob 模式（例如 <code>@doc/*.md</code> 或 <code>@.agent/**/*.md</code>）。</li>
</ul>
<h4>细粒度指导</h4>
<p>为了提供仅在处理特定文件时适用的指导，你可以在被提及文件的 YAML front matter 中指定 <code>globs</code>。</p>
<p>例如，要应用特定语言的编码规则：</p>
<ol>
<li>在你的 <code>AGENTS.md</code> 文件中的任意位置放置 <code>See @docs/*.md</code>。</li>
<li>创建一个文件 <code>docs/typescript-conventions.md</code>，内容如下：
<pre><code class="language-markdown">---
globs:
  - '**/*.ts'
  - '**/*.tsx'
---
Follow these TypeScript conventions:
- Never use the \`any\` type
- ...
</code></pre>
</li>
<li>对其他语言重复此步骤。</li>
</ol>
<p>只有当 Amp 读取了与任何 globs 匹配的文件（在上面的例子中是任何 TypeScript 文件）时，带有 <code>globs</code> 的被提及文件才会被包含。如果未指定 <code>globs</code>，则被 @提及的文件总是被包含。</p>
<p>Globs 隐式地以 <code>**/</code> 为前缀，除非它们以 <code>../</code> 或 <code>./</code> 开头，这种情况下它们指的是相对于被提及文件的路径。</p>
<p>其他示例：</p>
<ul>
<li>前端特定指导：<code>globs: ["src/components/**", "**/*.tsx"]</code></li>
<li>后端指导：<code>globs: ["server/**", "api/**"]</code></li>
<li>测试指导：<code>globs: ["*.test.ts", "__tests__/*"]</code></li>
</ul>
<h4>迁移到 AGENTS.md</h4>
<ul>
<li>从 Claude Code 迁移：<code>mv CLAUDE.md AGENTS.md &#x26;&#x26; ln -s AGENTS.md CLAUDE.md</code>，并对子树的 <code>CLAUDE.md</code> 文件重复此操作。</li>
<li>从 Cursor 迁移：<code>mv .cursorrules AGENTS.md &#x26;&#x26; ln -s AGENTS.md .cursorrules</code>，然后在 <code>AGENTS.md</code> 的任意位置添加 <code>@.cursor/rules/*.mdc</code> 以包含所有 Cursor 规则文件。</li>
<li>从现有的 AGENT.md 迁移：<code>mv AGENT.md AGENTS.md</code>（可选 - 两个文件名均可继续工作）。</li>
</ul>
<h3>接力 (Handoff)</h3>
<p>当你会话保持简短并专注于单一任务时，Amp 的效果最好。</p>
<p>要在新会话中继续你在旧会话中的工作，请使用命令面板中的 <code>handoff</code> 命令，从原始会话中提取相关文件和上下文来草拟一个新会话。</p>
<p>向 handoff 命令提供一些帮助以指导新提示词。例如：</p>
<ul>
<li><code>now implement this for teams as well, not just individual users</code>（现在也为团队实现这个，不仅仅是个人用户）</li>
<li><code>execute phase one of the created plan</code>（执行已创建计划的第一阶段）</li>
<li><code>check the rest of the codebase and find other places that need this fix</code>（检查代码库的其余部分，找到其他需要此修复的地方）</li>
</ul>
<p>关于 Amp 为什么不支持压缩 (compaction)，请参阅 <a href="https://ampcode.com/news/handoff">接力（不再压缩）</a>。</p>

<h3>编辑与撤销 (Edit &#x26; Undo)</h3>
<p>编辑会话中之前的消息会自动还原智能体在<em>该消息之后</em>所做的任何更改。</p>
<p>要在 CLI 中编辑之前的消息，请按 Tab 键导航到之前的消息。在编辑器扩展中，向上滚动会话并点击之前的消息。</p>

<ul>

<h3>自定义命令</h3>
<p>通过 Amp 命令面板访问自定义命令：在 VS Code/Cursor/Windsurf 中按 Cmd/Alt-Shift-A，或在 CLI 中按 Ctrl-O。</p>
<p>你可以创建自定义命令来复用提示词并自动化工作流。</p>
<p>要创建自定义命令，请在以下位置之一创建一个 Markdown 文件或可执行文件：</p>
<ul>
<li>当前工作区中的 <code>.agents/commands</code></li>
<li><code>~/.config/amp/commands</code>（如果设置了 <code>$XDG_CONFIG_HOME</code>，则使用该目录）</li>
</ul>
<p>这些目录中的每个文件，如果是以 <code>.md</code> 结尾的 Markdown 文件或可执行文件（设置了执行位或<a href="https://en.wikipedia.org/wiki/Shebang_(Unix)">第一行有 shebang</a>），都将转换为自定义命令。命令名称将是不带扩展名的文件名。</p>
<p>调用时，自定义命令将其输出追加到提示词输入中。Markdown 文件直接追加其内容。可执行文件运行并追加其组合的 stdout/stderr 输出（最多 5 万个字符）。可执行文件也可以接受参数，这些参数将在调用时传递。</p>
<p>以下是两个示例：</p>
<ul>
<li>文件 <code>.agents/commands/pr-review.md</code> 将转换为自定义命令 <code>pr-review</code>，并且 <code>pr-review.md</code> 的内容将插入到提示词输入中。</li>
<li><code>~/.config/amp/commands/outline</code> 是一个可执行文件，将转换为自定义命令 <code>outline</code>，并可以带参数使用：<code>outline src/utils</code></li>
</ul>
<p>实际案例：</p>
<ul>
<li><a href="https://github.com/nicolaygerold/howtobuildwithai/blob/main/.agents/commands/work-on-linear-issue">work-on-linear-issue</a></li>
<li><a href="https://github.com/nicolaygerold/howtobuildwithai/blob/main/.agents/commands/resolve-pr-comments">resolve-pr-comments</a></li>
<li><a href="https://github.com/nicolaygerold/howtobuildwithai/blob/main/.agents/commands/code-review.md">code-review.md</a></li>
</ul>

<h2>工具 (Tools)</h2>
<p>工具是底层模型用来辅助任务的东西。为了获得最高质量的结果，我们建议你使用一组精选的工具，并调整提示词以适应底层模型。</p>
<h3>内置工具</h3>
<p>你可以通过在 CLI 中运行 <code>amp tools list</code> 或在扩展的设置面板中查看 Amp 的内置工具。</p>
<h3>工具箱 (Toolboxes)</h3>
<p>工具箱允许你用简单的脚本扩展 Amp，而无需提供 MCP 服务器。</p>
<p>当 Amp 启动时，它会调用 <code>AMP_TOOLBOX</code> 指定目录中的每个可执行文件，并将环境变量 <code>TOOLBOX_ACTION</code> 设置为 <code>describe</code>。</p>
<p>该工具应将其描述作为键值对列表（每行一个）写入 <code>stdout</code>。</p>
<pre><code class="language-javascript">#!/usr/bin/env bun

const action = process.env.TOOLBOX_ACTION

if (action === 'describe') showDescription()
else if (action === 'execute') runTests()

function showDescription() {
    process.stdout.write(
        [
            'name: run-tests',
            'description: use this tool instead of Bash to run tests in a workspace',
            'dir: string the workspace directory',
        ].join('\n'),
    )
}
</code></pre>
<p>当 Amp 决定使用你的工具时，它会再次运行可执行文件，将 <code>TOOLBOX_ACTION</code> 设置为 <code>execute</code>。</p>
<p>工具以相同的格式在 <code>stdin</code> 上接收参数，然后执行其工作：</p>
<pre><code class="language-javascript">function runTests() {
    let dir = require('fs')
        .readFileSync(0, 'utf-8')
        .split('\n')
        .filter((line) => line.startsWith('dir: '))

    dir = dir.length > 0 ? dir[0].replace('dir: ', '') : '.'

    require('child_process').spawnSync('pnpm', ['-C', dir, 'run', 'test', '--no-color', '--run'], {
        stdio: 'inherit',
    })
}
</code></pre>
<p>如果你的工具需要对象或数组参数，可执行文件可以将 <a href="https://modelcontextprotocol.io/specification/2025-06-18/server/tools#tool">工具 schema</a> 作为 JSON 写入 <code>stdout</code>。在这种情况下，它也将接收 JSON 格式的输入。</p>
<p>我们建议使用工具来表达特定的、确定性的和项目本地的行为，例如：</p>
<ul>
<li>查询开发数据库，</li>
<li>在项目中运行测试和构建操作，</li>
<li>以受控方式暴露 CLI 工具。</li>
</ul>
<p>有关完整的技术参考，请参阅 <a href="https://ampcode.com/manual/appendix#toolboxes-reference">附录</a>。</p>
<h3>子智能体 (Subagents)</h3>
<p>Amp 可以（通过 Task 工具）生成子智能体来处理受益于独立执行的复杂任务。每个子智能体都有自己的上下文窗口，并能访问如文件编辑和终端命令等工具。</p>
<p>子智能体最适用于可以分解为独立部分的多步骤任务、产生完工后无需保留的大量输出的操作、跨不同代码区域的并行工作，以及在协调复杂工作时保持主会话上下文的整洁。</p>
<p>然而，子智能体是隔离工作的——它们无法相互通信，你无法在任务中途指导它们，它们从零开始而不带对话积累的上下文，并且主智能体只接收它们的最终总结，而不是监控它们的逐步工作。</p>
<p>Amp 可能会自动对合适的任务使用子智能体，或者你可以通过提及子智能体或建议并行工作来鼓励使用它们。</p>
<h3>先知 (Oracle)</h3>
<p>Amp 可以访问一个强大的“第二意见”模型，该模型更适合复杂的推理或分析任务，代价是速度稍慢、稍贵，且不如主智能体模型适合日常代码编辑任务。</p>
<p>Amp 的主智能体可以通过一个名为 <code>oracle</code> 的工具使用该模型，目前它使用的是 GPT-5，推理等级为中等（我们发现这在不花费过多时间的情况下效果很好）。</p>
<p>主智能体在调试或审查复杂代码时，可以自主决定向 Oracle 寻求帮助。由于成本较高且推理速度较慢，我们有意不强制主智能体<em>总是</em>使用 Oracle。</p>
<p>当你认为有帮助时，我们建议明确要求 Amp 的主智能体使用 Oracle。以下是我们自己使用 Amp 的一些示例：</p>
<ul>
<li>“使用 Oracle 审查上次提交的更改。我想确保空闲或需要用户输入时播放通知声音的实际逻辑没有改变。”</li>
<li>“询问 Oracle 是否有更好的解决方案。”</li>
<li>“我在这些文件中有个 bug：… 当我运行此命令时它会出现：… 帮我修复这个 bug。尽可能多地使用 Oracle，因为它很聪明。”</li>
<li>“分析函数 <code>foobar</code> 和 <code>barfoo</code> 是如何使用的。然后我想让你与 Oracle 大量合作，弄清楚我们如何重构它们之间的重复代码，同时保持更改的向后兼容性。”</li>
</ul>
<p>有关更多信息，请参阅 <a href="https://ampcode.com/news/gpt-5-oracle">GPT-5 Oracle 公告</a>。</p>
<h3>图书管理员 (Librarian)</h3>
<p>Amp 可以使用 Librarian 子智能体搜索远程代码库。Librarian 可以搜索和阅读 GitHub 上的所有公共代码以及你的私有 GitHub 仓库。</p>
<p>当你需要进行跨仓库研究，或者例如想要它阅读你正在使用的框架和库的代码时，告诉 Amp 召唤 Librarian。Librarian 的回答通常更长、更详细，因为我们将其构建为提供深入解释。Librarian 只会搜索仓库默认分支上的代码。</p>
<p>你需要在 <a href="https://ampcode.com/settings#code-host-connections">你的设置</a> 中配置 GitHub 连接才能使用它。如果你希望 Librarian 能看到你的私有仓库，你需要在配置 GitHub 连接时选择它们。有关更多信息，请参阅 GitHub 关于 <a href="https://docs.github.com/en/apps/using-github-apps/installing-a-github-app-from-a-third-party">安装</a> 和 GitHub Apps 的文档。</p>
<p>你可能需要明确提示主智能体使用 Librarian。以下是一些示例：</p>
<ul>
<li>“解释发布时如何部署我们文档的新版本。搜索我们的 docs 和 infra 仓库，看看它们是如何到达 X.Y.sourcegraph.com 的。”</li>
<li>“我这个使用 Zod 的验证代码有个 bug，它抛出了一个奇怪的错误。请 Librarian 调查为什么会发生这个错误，并向我展示导致它的逻辑。”</li>
<li>“使用 Librarian 调查 <code>foo</code> 服务——我在 <code>bar</code> 中使用的 API 端点最近有变更吗？如果有，变了什么，什么时候合并的？”</li>
</ul>
<p>有关更多信息，请参阅 <a href="https://ampcode.com/news/librarian">Librarian 公告</a>。</p>
<h3>MCP</h3>
<p>你可以使用 <a href="https://modelcontextprotocol.io/">MCP (Model Context Protocol)</a> 服务器添加额外的工具，这些服务器可以是本地的也可以是远程的。可以在 <a href="https://ampcode.com/#configuration">配置文件</a> 中的 <code>amp.mcpServers</code> 中配置它们。你也可以在 VS Code 的设置下按 + Add MCP Server。</p>
<p>本地 MCP 服务器的配置选项：</p>
<ul>
<li><code>command</code> - 可执行文件</li>
<li><code>args</code> - 命令参数（可选）</li>
<li><code>env</code> - 环境变量（可选）</li>
</ul>
<p>远程 (HTTP/SSE) MCP 服务器的配置选项：</p>
<ul>
<li><code>url</code> - 服务器端点</li>
<li><code>headers</code> - 发送请求时带的 HTTP 头（可选）</li>
</ul>
<p>Amp 会根据服务器的响应头自动检测适当的传输类型（HTTP 或 SSE）。</p>
<p>有两种方法可以验证 MCP 服务器：</p>
<ul>
<li>如果远程 MCP 服务器需要授权，你可以通过 <code>headers</code> 选项直接传递身份验证头。</li>
<li>对于 OAuth 身份验证，请使用 Amp 内置的 OAuth 支持（参见下文 <a href="https://ampcode.com/#mcp-oauth">远程 MCP 服务器的 OAuth</a>）。</li>
</ul>
<p>你也可以在配置中使用 <code>${VAR_NAME}</code> 语法的环境变量。</p>
<p>配置示例：</p>
<pre><code class="language-json">"amp.mcpServers": {
    "playwright": {
        "command": "npx",
        "args": ["-y", "@playwright/mcp@latest", "--headless", "--isolated"]
    },
    "semgrep": {
        "url": "https://mcp.semgrep.ai/mcp"
    },
    "sourcegraph": {
        "url": "${SRC_ENDPOINT}/.api/mcp/v1",
        "headers": {
            "Authorization": "token ${SRC_ACCESS_TOKEN}"
        }
    },
    "linear": {
        "command": "npx",
        "args": [
            "mcp-remote",
            "https://mcp.linear.app/sse"
        ]
    },
    "monday": {
        "url": "https://mcp.monday.com/sse",
        "headers": {
            "Authorization": "Bearer ${MONDAY_API_TOKEN}"
        }
    }
}
</code></pre>
<p>你也可以使用 CLI 添加带有 header 选项的 MCP 服务器：</p>
<pre><code>$ amp mcp add sourcegraph --header "Authorization=token sgp_your-token-here" https://sourcegraph.example.com/.api/mcp/v1
</code></pre>
<p>过多的可用工具可能会降低模型性能，因此为了获得最佳结果，请进行筛选：</p>
<ul>
<li>使用暴露少量高质量工具且描述清晰的 MCP 服务器。</li>
<li>禁用你不使用的 MCP 工具，方法是在扩展的设置界面中悬停在工具名称上并点击，使其显示为 <del>tool_name</del>，或者将它们添加到 <a href="https://ampcode.com/#configuration">配置文件</a> 中的 <code>amp.tools.disable</code>。</li>
<li>考虑使用 CLI 工具代替 MCP 服务器。</li>
</ul>
<h4>远程 MCP 服务器的 OAuth</h4>
<p>Amp 支持远程 MCP 服务器的 OAuth 身份验证。有两种可用的身份验证流程：</p>
<p><strong>动态客户端注册 (DCR)</strong></p>
<p>一些 MCP 服务器如 <a href="https://linear.app/changelog/2025-05-01-mcp">Linear</a> 支持自动 OAuth 客户端注册。当你添加此类服务器时，Amp 会在启动时自动在浏览器中开始 OAuth 流程：</p>
<pre><code>$ amp mcp add linear https://mcp.linear.app/sse
</code></pre>
<p><strong>手动 OAuth 客户端注册</strong></p>
<p>对于需要手动配置 OAuth 客户端的服务器：</p>
<ol>
<li>在服务器的管理界面中创建一个 OAuth 客户端，配置如下：
<ul>
<li>重定向 URI: <code>http://localhost:8976/oauth/callback</code></li>
<li>你的用例所需的 scopes</li>
</ul>
</li>
<li>将 MCP 服务器添加到你的配置中：</li>
</ol>
<pre><code>$ amp mcp add my-server https://example.com/.api/mcp/v1
</code></pre>
<ol>
<li>注册你的 OAuth 凭据：</li>
</ol>
<pre><code>$ amp mcp oauth login my-server 
  --server-url https://example.com/.api/mcp/v1 
  --client-id your-client-id 
  --client-secret your-client-secret 
  --scopes "openid,profile,email,user:all"
</code></pre>
<p>启动时，Amp 将打开浏览器以完成身份验证流程。</p>
<p>OAuth 令牌安全地存储在 <code>~/.amp/oauth/</code> 中，并在需要时自动刷新。</p>
<h3>权限 (Permissions)</h3>
<p>在调用工具之前，Amp 会检查用户的权限列表，以第一个匹配的条目来决定是否运行该工具。</p>
<p>如果未找到匹配项，Amp 会扫描其内置的权限列表，如果在那里也未找到匹配项，则拒绝使用该工具。</p>
<p>匹配的条目告诉 Amp 要么<em>允许</em>使用工具而无需询问，要么直接<em>拒绝</em>使用工具，要么<em>询问</em>操作员，要么将决定<em>委派</em>给另一个程序。</p>
<p>权限在你的 <a href="https://ampcode.com/#configuration">配置文件</a> 中的 <code>amp.permissions</code> 条目下配置：</p>
<pre><code class="language-json">"amp.permissions": [
  // 运行包含 git commit 的命令行前询问
  { "tool": "Bash", "matches": { "cmd": "*git commit*" }, "action": "ask"},
  // 拒绝包含 python or python3 的命令行
  { "tool": "Bash", "matches": { "cmd": ["*python *", "*python3 *"] }, "action": "reject"},
  // 允许所有 playwright MCP 工具
  { "tool": "mcp__playwright_*", "action": "allow"},
  // 运行任何其他 MCP 工具前询问
  { "tool": "mcp__*", "action": "ask"},
  // 将其他所有情况委派给权限助手（必须在 $PATH 中）
  { "tool": "*", "action": "delegate", "to": "my-permission-helper"}
]
</code></pre>
<h4>在 VS Code 中使用权限</h4>
<p>复杂对象必须在 VS Code 的设置 JSON 中配置。</p>
<p>VS Code 中集成了权限的 JSON schema，以便在编辑权限时提供指导。</p>
<p>动作为 <code>ask</code> 的规则在 VS Code 中仅对 <code>Bash</code> 工具有效。</p>
<h4>在 CLI 中使用权限</h4>
<p>使用 <code>amp permissions edit</code>，你可以使用 <code>$EDITOR</code> 编程和交互式地编辑你的权限规则。</p>
<p><code>amp permissions test</code> 命令评估权限规则而不实际运行任何工具，为验证你的规则是否按预期工作提供了一种安全的方式。</p>
<pre><code>$ amp permissions edit &#x3C;&#x3C;'EOF'
allow Bash --cmd 'git status' --cmd 'git diff*'
ask Bash --cmd '*'
EOF
$ amp permission test Bash --cmd 'git diff --name-only'
tool: Bash
arguments: {"cmd":"git diff --name-only"}
action: allow
matched-rule: 0
source: user
$ amp permission test Bash --cmd 'git push'
tool: Bash
arguments: {"cmd":"push"}
action: ask
matched-rule: 1
source: user
</code></pre>
<p>运行 <code>amp permissions list</code> 以 <code>amp permissions edit</code> 能理解的相同格式显示已知权限规则：</p>
<pre><code>$ amp permissions list
allow Bash --cmd 'git status' --cmd 'git diff*'
ask Bash --cmd '*'
</code></pre>
<p>请参阅 <code>amp permissions --help</code> 的输出以获取可用操作的完整集合。</p>
<h4>将权限决策委派给外部程序</h4>
<p>为了完全控制，你可以告诉 Amp 在调用工具之前咨询另一个程序：</p>
<pre><code class="language-json">{ "action": "delegate", "to": "amp-permission-helper", "tool": "Bash" }
</code></pre>
<p>现在，每当 Amp 想要运行 shell 命令时，它都会调用 <code>amp-permission-helper</code>：</p>
<pre><code class="language-python">#!/usr/bin/env python3
import json, sys, os

tool_name = os.environ.get("AGENT_TOOL_NAME")
tool_arguments = json.loads(sys.stdin.read())

# allow all other tools
if tool_name != "Bash":
    sys.exit(0)

# reject git push outright - stderr is passed to the model
if 'git push' in tool_arguments.get('cmd', ''):
    print("Output the correct command line for pushing changes instead", file=sys.stderr)
    sys.exit(2)

# ask in any other case
sys.exit(1)
</code></pre>
<p>错误代码和 stderr 用于告诉 Amp 如何继续。</p>
<p>有关完整的技术参考，请参阅 <a href="https://ampcode.com/manual/appendix#permissions-reference">附录</a>。</p>
<h2>会话共享 (Thread Sharing)</h2>
<p>会话（Threads）是你与智能体的对话，包含所有消息、上下文和工具调用。你的会话可在 <a href="https://ampcode.com/threads">ampcode.com/threads</a> 查看。</p>
<p>我们发现在代码审查中包含 Amp 会话链接非常有用，可以给审查者提供更多背景信息。阅读和搜索团队的会话也可以帮助你了解正在发生的事情以及其他人如何使用 Amp。</p>
<p>要更改会话的共享对象：</p>
<ul>
<li>在 CLI 中，输入 / 呼出命令面板，然后选择 <code>thread: set visibility</code>。</li>
<li>在编辑器扩展或网页上，使用顶部的共享菜单。</li>
</ul>
<p>会话的可见性级别可以设置为：</p>
<ul>
<li>公开 (Public)：对你的公开个人资料 (<code>ampcode.com/@*your-username*</code>) 上的任何人可见，且可公开搜索</li>
<li>不列出 (Unlisted)：对拥有链接的互联网上任何人可见，并在工作区内共享</li>
<li>工作区共享 (Workspace-shared)：对工作区的所有成员可见</li>
<li>组共享 (Group-shared)：对你选择的特定组成员可见；仅限企业工作区</li>
<li>私有 (Private)：仅对你可见</li>
</ul>
<p>如果你不在工作区中，默认情况下你的会话仅对你可见。</p>
<p>如果你在工作区中，默认情况下你的会话将与工作区成员共享。<a href="https://ampcode.com/#enterprise">企业 (Enterprise)</a> 工作区可以配置额外的共享控制；参见 <a href="https://ampcode.com/manual/appendix#workspace-thread-visibility-controls">工作区会话可见性控制</a>。</p>
<h2>CLI</h2>
<p><a href="https://ampcode.com/#getting-started-command-line-interface">安装</a> 并登录后，运行 <code>amp</code> 启动 Amp CLI。</p>
<p>不带任何参数时，它以交互模式运行：</p>
<pre><code>$ amp
</code></pre>
<p>如果你将输入通过管道传输到 CLI，它会将输入用作交互模式下的第一条用户消息：</p>
<pre><code>$ echo "commit all my changes" | amp
</code></pre>
<p>使用 <code>-x</code> 或 <code>--execute</code> 以执行模式启动 CLI（在 <a href="https://ampcode.com/free">Amp Free</a> 中不可用）。在此模式下，它将 <code>-x</code> 提供的消息发送给智能体，等待智能体结束其回合，打印最终消息并退出：</p>
<pre><code>$ amp -x "what files in this folder are markdown files? Print only the filenames."
README.md
AGENTS.md
</code></pre>
<p>使用 <code>-x</code> 时也可以通过管道传输输入：</p>
<pre><code>$ echo "what package manager is used here?" | amp -x
cargo
</code></pre>
<p>如果你想使用 <code>-x</code> 且智能体使用的工具可能需要批准，请务必使用 <code>--dangerously-allow-all</code> 或 <a href="https://ampcode.com/#permissions">配置 Amp 允许它们</a>：</p>
<pre><code>$ amp --dangerously-allow-all -x "Run \`sed\` to replace 2024 with 2025 in README."
Done. Replaced 8 occurrences of 2024 in README.md
</code></pre>
<p>重定向 stdout 时，会自动开启执行模式：</p>
<pre><code>$ echo "what is 2+2?" | amp > response.txt
</code></pre>
<p>当你通过管道传输输入并用 <code>-x</code> 提供提示词时，智能体可以看到两者：</p>
<pre><code>$ cat ~/.vimrc | amp -x "which colorscheme is used?"
The colorscheme used is **gruvbox** with dark background and hard contrast.

\`\`\`vim
set background=dark
let g:gruvbox_contrast_dark = "hard"
colorscheme gruvbox
\`\`\`
</code></pre>
<p>你可以将 <code>--mcp-config</code> 标志与 <code>-x</code> 命令一起使用，以指定 MCP 服务器而无需修改配置文件。</p>
<pre><code>$ amp --mcp-config '{"everything": {"command": "npx", "args": ["-y", "@modelcontextprotocol/server-everything"]}}' -x "What tools are available to you?"
</code></pre>
<p>要查看 CLI 的更多功能，请运行 <code>amp --help</code>。</p>
<h3>非交互环境</h3>
<p>对于非交互环境（例如脚本、CI/CD 管道），请将你的 <a href="https://ampcode.com/settings">API 密钥</a> 设置在环境变量中：</p>
<pre><code class="language-bash">export AMP_API_KEY=your-api-key-here
</code></pre>
<h3>CLI–IDE 集成</h3>
<p>Amp CLI 与 VS Code、JetBrains 和 Neovim 集成（安装请见 <a href="https://ampcode.com/install">ampcode.com/install</a>），这使得 Amp CLI 能够：</p>
<ul>
<li>读取诊断信息，例如类型检查器和 linter 错误</li>
<li>查看当前打开的文件和选区，以便 Amp 更好地理解你的提示词上下文</li>
<li>通过 IDE 编辑文件，并支持完全撤销</li>
</ul>
<p>CLI 在大多数情况下会自动检测你是否运行了 Amp 编辑器扩展。如果你使用 JetBrains 并在 JetBrains 内置终端<em>以外</em>的终端运行 Amp CLI，则需要运行 <code>amp --jetbrains</code> 来检测它。</p>
<h3>Shell 模式</h3>
<p>通过以 <code>$</code> 开头输入消息，直接在 CLI 中执行 shell 命令。命令及其输出将包含在下一条发给智能体的消息的上下文窗口中。</p>
<p>使用 <code>$$</code> 激活隐身 shell 模式，命令会执行但不会包含在上下文中。这对于嘈杂的命令或你通常会在单独终端中运行的快速检查很有用。</p>
<h3>在 CLI 中编写提示词</h3>
<p>在现代终端模拟器中，如 Ghostty、Wezterm、Kitty 或 iTerm2，你可以使用 shift-enter 在提示词中插入换行符。</p>
<p>此外，你还可以键入 <code>\</code> 后跟回车键来插入换行符。</p>
<p>如果你设置了环境变量 <code>$EDITOR</code>，可以使用命令面板中的 <code>editor</code> 命令打开编辑器来编写提示词。</p>
<h3>流式 JSON (Streaming JSON)</h3>
<p>Amp 的 CLI 支持流式 JSON 输出格式，stdout 每行一个对象，用于编程集成和实时对话监控。</p>
<p>使用 <code>--stream-json</code> 标志配合 <code>--execute</code> 模式，以流式 JSON 格式输出而非纯文本。</p>
<p>带参数的基本用法：</p>
<pre><code>$ amp --execute "what is 3 + 5?" --stream-json
</code></pre>
<p>结合 <code>--stream-json</code> 和 <code>amp threads continue</code>：</p>
<pre><code>$ amp threads continue --execute "now add 8 to that" --stream-json
</code></pre>
<p>使用 stdin 输入：</p>
<pre><code>$ echo "analyze this code" | amp --execute --stream-json
</code></pre>
<p>你可以在 <a href="https://ampcode.com/manual/appendix?preview#message-schema">附录中找到 JSON 输出的 schema</a>。</p>
<p>也可以使用 <code>--stream-json-input</code> 标志在 stdin 上提供输入：</p>
<pre><code>$ echo '{
  "type": "user",
  "message": {
    "role": "user",
    "content": [
      {
        "type": "text",
        "text": "what is 2+2?"
      }
    ]
  }
}' | amp -x --stream-json --stream-json-input
</code></pre>
<p><code>--stream-json</code> 标志需要 <code>--execute</code> 模式。它不能单独使用。而 <code>--stream-json-input</code> 需要 <code>--stream-json</code>。</p>
<p>当使用 <code>--stream-json-input</code> 时，<code>--execute</code> 的行为会发生变化，Amp 只有在助手完成<em>并且</em> stdin 已关闭时才会退出。</p>
<p>这允许编程方式使用 Amp CLI 进行包含多条用户消息的对话。</p>
<pre><code class="language-bash">#!/usr/bin/env bash

send_message() {
  local text="$1"
  echo '{"type":"user","message":{"role":"user","content":[{"type":"text","text":"'$text'"}]}}'
}

{
  send_message "what's 2+2?"
  sleep 10

  send_message "now add 8 to that"
  sleep 10

  send_message "now add 5 to that"
} | amp --execute --stream-json --stream-json-input
</code></pre>
<p>有关输出的 schema、示例输出和更多用法示例，请参阅 <a href="https://ampcode.com/manual/appendix#stream-json-output">附录</a>。</p>
<h2>配置 (Configuration)</h2>
<p>Amp 可以通过编辑器扩展中的设置（例如 <code>.vscode/settings.json</code>）和 CLI 配置文件进行配置。</p>
<p>CLI 配置文件位置因操作系统而异：</p>
<ul>
<li>macOS: <code>~/.config/amp/settings.json</code></li>
<li>Linux: <code>~/.config/amp/settings.json</code></li>
<li>Windows: <code>%APPDATA%\amp\settings.json</code></li>
</ul>
<p>所有设置都使用 <code>amp.</code> 前缀。</p>
<h3>设置</h3>
<h4>编辑器扩展和 CLI</h4>
<ul>
<li><strong><code>amp.anthropic.thinking.enabled</code></strong>
<strong>类型:</strong> <code>boolean</code>, <strong>默认值:</strong> <code>true</code>
启用 Claude 的扩展思考能力</li>
<li><strong><code>amp.experimental.dojoMode</code></strong>
<strong>类型:</strong> <code>boolean</code>, <strong>默认值:</strong> <code>false</code>
启用道场模式 (dojo mode)——一种学习模式，智能体通过苏格拉底式提问引导你自己发现解决方案。智能体可以阅读你的代码库并搜索信息，但不会编辑文件或给出直接解决方案。非常适合学习编程或理解新的代码库。</li>
<li><strong><code>amp.experimental.planMode</code></strong>
<strong>类型:</strong> <code>boolean</code>, <strong>默认值:</strong> <code>false</code>
启用计划模式 (plan mode)，以便在执行前与智能体进行协作规划。在计划模式下，智能体可以研究你的代码库，但不能编辑文件或运行命令。</li>
<li><strong><code>amp.permissions</code></strong>
<strong>类型:</strong> <code>array</code>, <strong>默认值:</strong> <code>[]</code>
配置允许、拒绝或询问批准哪些工具的使用。参见 <a href="https://ampcode.com/#permissions">权限</a>。</li>
<li><strong><code>amp.tab.clipboard.enabled</code></strong>
<strong>类型:</strong> <code>boolean</code>, <strong>默认值:</strong> <code>true</code>
启用 Amp Tab 上下文的剪贴板访问</li>
<li><strong><code>amp.git.commit.ampThread.enabled</code></strong>
<strong>类型:</strong> <code>boolean</code>, <strong>默认值:</strong> <code>true</code>
启用在 git 提交中添加 Amp-Thread trailer。禁用时，使用提交工具进行的提交将不包含 <code>Amp-Thread: &#x3C;thread-url></code> trailer。</li>
<li><strong><code>amp.git.commit.coauthor.enabled</code></strong>
<strong>类型:</strong> <code>boolean</code>, <strong>默认值:</strong> <code>true</code>
启用在 git 提交中将 Amp 添加为共同作者 (co-author)。禁用时，使用提交工具进行的提交将不包含 <code>Co-authored-by: Amp &#x3C;amp@ampcode.com></code> trailer。</li>
<li><strong><code>amp.mcpServers</code></strong>
<strong>类型:</strong> <code>object</code>
暴露工具的 Model Context Protocol 服务器。参见 <a href="https://ampcode.com/#mcp">自定义工具 (MCP) 文档</a>。</li>
<li><strong><code>amp.model.sonnet</code></strong>
<strong>类型:</strong> <code>boolean</code>, <strong>默认值:</strong> <code>false</code>
<em>临时设置。</em> 使用 Claude Sonnet 4.5 代替 Gemini 3 作为主要模型。提供此设置是为了帮助过渡到 Gemini 3，并将在未来的版本中删除。也可以通过 CLI 标志启用：<code>amp --use-sonnet</code>。</li>
<li><strong><code>amp.terminal.commands.nodeSpawn.loadProfile</code></strong>
<strong>类型:</strong> <code>string</code>, <strong>默认值:</strong> <code>"always"</code>, <strong>选项:</strong> <code>"always"</code> | <code>"never"</code> | <code>"daily"</code>
在运行命令（包括 MCP 服务器）之前，是否加载用户 profile (<code>.bashrc</code>, <code>.zshrc</code>, <code>.envrc</code>) 中的环境变量（从工作区根目录可见）</li>
<li><strong><code>amp.todos.enabled</code></strong>
<strong>类型:</strong> <code>boolean</code>, <strong>默认值:</strong> <code>true</code>
启用用于管理任务的 TODO 跟踪</li>
<li><strong><code>amp.tools.disable</code></strong>
<strong>类型:</strong> <code>array</code>, <strong>默认值:</strong> <code>[]</code>
按名称禁用特定工具。使用 ‘builtin:toolname’ 仅禁用具有该名称的内置工具（允许 MCP 服务器提供同名工具）。支持使用 <code>*</code> 的 Glob 模式。</li>
<li><strong><code>amp.tools.stopTimeout</code></strong>
<strong>类型:</strong> <code>number</code>, <strong>默认值:</strong> <code>300</code>
在取消正在运行的工具之前等待多少秒</li>
<li><strong><code>amp.mcpPermissions</code></strong>
<strong>类型:</strong> <code>array</code>, <strong>默认值:</strong> <code>[]</code>
允许或阻止匹配指定模式的 MCP 服务器。应用第一个匹配的规则。如果没有规则匹配 MCP 服务器，该服务器将被允许。
<ul>
<li><strong>远程 MCP 服务器</strong>: 使用 <code>url</code> 键为服务器端点指定匹配标准</li>
<li><strong>本地 MCP 服务器</strong>: 使用 <code>command</code> 和 <code>args</code> 键匹配可执行命令及其参数
以下是一些示例：</li>
</ul>
<pre><code class="language-json">"amp.mcpPermissions": [
  // 允许特定的受信任 MCP 服务器
  { "matches": { "command": "npx", "args": "* @playwright/mcp@*" }, "action": "allow" },
  { "matches": { "url": "https://mcp.trusted.com/mcp" }, "action": "allow" },
  // 阻止潜在风险的 MCP 服务器
  { "matches": { "command": "python", "args": "*bad_command*" }, "action": "reject" },
  { "matches": { "url": "*/malicious.com*" }, "action": "reject" },
]
</code></pre>
以下规则将阻止所有 MCP 服务器：
<pre><code class="language-json">"amp.mcpPermissions": [
  { "matches": { "command": "*" }, "action": "reject" },
  { "matches": { "url": "*" }, "action": "reject" }
]
</code></pre>
</li>
</ul>
<h4>仅限 CLI</h4>
<ul>
<li><strong><code>amp.updates.mode</code></strong>
<strong>类型:</strong> <code>string</code>, <strong>默认值:</strong> <code>"auto"</code>
控制更新检查行为：<code>"warn"</code> 显示更新通知，<code>"disabled"</code> 关闭检查，<code>"auto"</code> 自动运行更新。注意：设置 <code>AMP_SKIP_UPDATE_CHECK=1</code> 环境变量将覆盖此设置并禁用所有更新检查。</li>

<h3>企业托管设置</h3>
<p><a href="https://ampcode.com/#enterprise">企业 (Enterprise)</a> 工作区管理员可以通过将策略部署到运行 Amp 的机器上的以下位置，来强制执行覆盖用户和工作区设置的配置：</p>
<ul>
<li><strong>macOS</strong>: <code>/Library/Application Support/ampcode/managed-settings.json</code></li>
<li><strong>Linux</strong>: <code>/etc/ampcode/managed-settings.json</code></li>
<li><strong>Windows</strong>: <code>C:\ProgramData\ampcode\managed-settings.json</code></li>
</ul>
<p>此托管设置文件使用与 <a href="https://ampcode.com/#core-settings">常规设置</a> 文件相同的 schema，但多了一个字段：</p>
<p>amp.admin.compatibilityDate <code>string</code></p>
<h3>代理和证书</h3>
<p>在具有代理服务器或自定义证书的公司网络中使用 Amp CLI 时，请根据需要在 shell profile 或 CI 环境中设置这些标准 Node.js 环境变量：</p>
<pre><code class="language-bash">export HTTP_PROXY=your-proxy-url
export HTTPS_PROXY=your-proxy-url
export NODE_EXTRA_CA_CERTS=/path/to/your/certificates.pem
</code></pre>
<h2>定价</h2>
<h3>免费 (Free)</h3>
<p>Amp 的 <code>free</code> 模式是免费的，并由广告支持。它混合使用了顶级 OSS 模型、上下文窗口受限的前沿模型以及测试中的预发布前沿模型。</p>
<p><code>free</code> 模式符合 Amp 付费智能模式的所有严格 <a href="https://ampcode.com/security">安全标准</a>。你不必共享你的数据用于训练。</p>
<p>要使用它：在 Amp CLI 中输入 <code>/mode free</code>，或在 Amp 编辑器扩展的提示词字段中选择 <code>free</code> 模式（而不是付费的 <code>smart</code> 模式）。</p>
<p>有关更多信息，请参阅 <a href="https://ampcode.com/news/amp-free">Amp Free 公告</a> 和 <a href="https://ampcode.com/news/amp-free-no-training">在工作中使用 Amp Free</a>。</p>
<h3>付费使用</h3>
<p>Amp 默认的 <code>smart</code> 模式比 <code>free</code> 更自主、更强大，并且使用付费额度（credits）。</p>
<p>你可以在 <a href="https://ampcode.com/settings">用户设置</a> 中为自己购买更多额度，或者在 <a href="https://ampcode.com/workspace">工作区设置</a> 中为你的团队购买。注册后，大多数用户会收到 10 美元的免费额度。</p>
<p>使用量是根据 LLM 使用量和某些其他产生服务成本的工具（如网络搜索）的使用量来消耗的。对于个人和非企业工作区，我们直接按成本价向你收取这些费用，不加价。</p>
<p>工作区额度由所有工作区成员汇集并共享。所有未使用的额度在账户不活动一年后过期。</p>
<h3>企业版 (Enterprise)</h3>
<p>企业版的使用费比个人和团队计划贵 50%，并包括以下权限：</p>
<ul>
<li>SSO (Okta, SAML 等) 和目录同步</li>
<li>LLM 推理中的文本输入零数据保留</li>
<li>高级 <a href="https://ampcode.com/manual/appendix#workspace-thread-visibility-controls">会话可见性控制</a></li>
<li><a href="https://ampcode.com/#enterprise-managed-policy-settings">托管用户设置</a></li>
<li>用于工作区分析和数据管理的 API</li>
<li>可配置的会话保留期（按需提供）</li>
<li>用于工作区访问的 IP 白名单（按需提供）</li>
</ul>
<p>有关 Amp 企业版安全功能的更多信息，请参阅 <a href="https://ampcode.com/security">Amp 安全参考</a>。</p>
<p>要开始使用 Amp 企业版，请前往 <a href="https://ampcode.com/team">你的工作区</a> 并点击右上角的“更改计划 (Change Plan)”。这需要一次性购买 1,000 美元，这将授予你的工作区 1,000 美元的 Amp 企业版使用额度，并将你的工作区升级为企业版。</p>
<p>使用 <strong>Amp Enterprise Premium</strong>，5,000 美元以上的购买可提供发票支付，25,000 美元以上的购买可享受批量折扣。Amp Enterprise Premium 还包括以下权限：</p>
<ul>
<li>用于每用户成本控制的 <a href="https://ampcode.com/manual/appendix#workspace-entitlements">Entitlements</a></li>
<li>用于成本归因和每组会话可见性选项的用户组（按需提供）</li>
</ul>
<p>请联系 <a href="https://ampcode.com/">amp-devs@sourcegraph.com</a> 以获取这些购买选项的访问权限以及有关 Amp 企业版的一般信息。</p>
