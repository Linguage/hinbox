3:T3e34,<h1>Zed是我们的办公室</h1>
<ul>
<li>来源：<a href="https://zed.dev/blog/zed-is-our-office">Zed Is Our Office</a></li>
</ul>
<p>现在是美东时间周一中午 12 点，整个 Zed Industries 团队都挤进了我们每周的“全员大会”（All-hands）。有些队友正在草草记下他们的日程变动，而其他人则在详细列出本周的关注重点。<a href="https://github.com/nathansobo">Nathan</a> 刚刚结束了首要事项的公告，<a href="https://github.com/morgankrey">Morgan</a> 正在分享我们的各项指标趋势以及运营更新。与此同时，我正在准备上周的用户语录以便分享，其他人则在向 <code>Discussions</code>（讨论）部分添加话题。</p>
<p>在整个会议过程中，屏幕被共享，各种声音在对话中切入切出，随着几十个光标在同一文件中实时并发编辑，我们的会议笔记也在飞速增长。</p>
<p><strong>这整个会议完全是在 Zed 内部进行的。</strong></p>
<p><img src="https://zed.dev/img/post/zed-is-our-office/this-week.webp" alt=""></p>
<p>从一开始，我们的使命就是打造一款具备以下特性的编辑器：</p>
<ol>
<li><strong><a href="https://zed.dev/blog/videogame">响应迅速</a></strong>：按键与重新渲染之间的延迟应当是难以察觉的。</li>
<li><strong>专注</strong>：界面应提供最少的干扰，不阻碍代码编写。</li>
<li><strong>协作</strong>：与队友一起工作的感觉应该和坐在办公室里并肩作战没有什么不同。</li>
</ol>
<p>抛开前两点不谈，让我们专注于<strong>协作</strong>。</p>
<h2>刻入 Zed 基因的协作能力</h2>
<p>多年来，我们一直梦想着构建终极的协作编辑器。这一愿景的根源可以追溯到 Nathan 在 <a href="https://en.wikipedia.org/wiki/Pivotal_Labs">Pivotal Labs</a> 的早期岁月，那里<a href="https://www.wired.com/2013/11/pivotal-one/">将两个键盘插入同一台电脑进行结对编程</a>是标准做法。我们要为分布式团队重现那种无缝的协作体验。</p>
<blockquote>
<p>等等……其他编辑器不是已经有这种技术了吗？</p>
</blockquote>
<p>是的！如果你做开发的时间足够长，你可能还记得 <a href="https://atom-editor.cc/">Atom</a> 的 <a href="https://atom-editor.cc/teletype">teletype</a> 插件——这两者都是由 Zed 的创始人构建的。Teletype 允许开发者分享进入他们工作区的“传送门”，这是迈向 Zed 协作愿景的第一步。尽管团队<a href="https://atom-editor.cc/blog/2017/08/08/atom-1-19#improved-responsiveness-and-memory-usage">尝试</a>让 Atom（一个 <a href="https://www.electronjs.org/">Electron</a> 应用）响应更迅速，但它从未达到团队渴望的性能标准。Nathan 离开了 Atom 团队，最终开始致力于 <a href="https://www.gpui.rs/">gpui</a> 的开发，这是 Zed 基于 Rust 编写的 GPU 加速 UI 渲染框架。之后 Atom 被 <a href="https://github.blog/news-insights/product-news/sunsetting-atom/">GitHub 关停</a>。没有了 Atom，也就没有了 Teletype。</p>
<p>其他编辑器也添加了各自版本的协作功能，但整体体验仍有不足。设置过程繁琐得令人烦恼；你通常需要安装扩展，而且每次想要分享时都得把链接粘贴到终端或编辑器中。并发编辑无法干净地合并，随着更多协作者的加入，性能会迅速下降。最糟糕的是，你往往最终还是得通过 Slack 或 Zoom 共享屏幕。</p>
<p>我们在设计 Zed 时，从底层就开始考虑协作——它<strong>绝非</strong>生搬硬套的服务，也<strong>不是</strong>后知后觉的补救。</p>
<p>利用 <a href="https://zed.dev/blog/crdts">CRDTs</a>（无冲突复制数据类型）作为我们的核心数据结构，我们确保了无冲突和最终一致性的属性，所有人的更改都能无缝合并并收敛到相同的状态。你不需要为了避免协作服务的致命缺陷而在光标操作上小心翼翼。无论同事是在同一间办公室还是在大洋彼岸，我们的架构都能提供低延迟，而且无论是结对编程（Pair Programming）还是群体编程（Mob Programming），性能都保持轻快。</p>
<p>设置也是毫不费力的：无需安装扩展，无需复制粘贴每次会话的链接；只需要你的 GitHub 账号。借助内置的语音功能和自动切换的屏幕共享，当你需要交流编辑器之外的工作时，无需退回到外部工具。</p>
<p>我们构建 Zed 的协作服务主要是为了我们自己，这样我们可以通过 Zed，在 Zed 中有效地一起构建 Zed。这不仅是一个功能，这对我们的工作方式<strong>至关重要</strong>。我们从中受益匪浅，并在此过程中找到了巨大的乐趣，我们相信你也会喜欢！</p>
<h2>Zed 协作工具速览</h2>
<p><img src="https://zed.dev/img/post/zed-is-our-office/collab-annotated.webp" alt=""></p>
<ol>
<li>通过点击状态栏中的人物图标打开协作面板，通过 GitHub 认证流程登录后即可访问。</li>
<li>该区域包含被称为“频道（Channel）”的虚拟房间，按层级结构组织。</li>
<li>点击 <code>+</code> 按钮创建顶级频道。右键点击现有频道并选择 <code>New Subchannel</code>（新建子频道）选项来创建嵌套的子频道。</li>
<li>GitHub 头像显示谁在哪个频道中。点击频道名称即可加入。</li>
<li>点击文档图标访问其“频道笔记”，它作为与频道关联的元数据。</li>
<li>进入频道后，通过麦克风图标开启/静音你的语音。</li>
<li>允许他人查看你的屏幕。</li>
<li>频道是<strong>与项目无关的</strong>。项目是通过标题栏中的 <code>Share</code>（分享）按钮在频道<em>内</em>自愿分享的。频道可以是公开的 (🛜) 或仅限特定成员 (#️⃣)，并包含 <code>Guest</code>（访客）、<code>Member</code>（成员）和 <code>Admin</code>（管理员）角色的权限系统。</li>
<li>点击标题栏中的头像以跟随队友。如果你正在跟随的人正在共享屏幕，Zed 会自动在跟随他们 Zed 实例中的光标和他们的屏幕共享之间切换，具体取决于他们是聚焦在 Zed 上还是其他应用程序上。</li>
</ol>
<p><em>有关协作的数据和隐私问题，请参阅我们的 <a href="https://zed.dev/faq#data-and-privacy">常见问题解答 (FAQs)</a>。</em></p>
<h2>我们的虚拟办公室</h2>
<p>我们的办公室就是 Zed 的协作面板。</p>
<p><img src="https://zed.dev/img/post/zed-is-our-office/tree-focus.webp" alt=""></p>
<p>随着公司的成长，<a href="https://zed.dev/channel/zed-283">我们的频道树</a> 经历了多次迭代，但今天的结构足够灵活，可以适应多种形式的协作。我们的频道树用于：</p>
<ul>
<li>全公司范围的讨论</li>
<li>项目工作</li>
<li>个人专注时间</li>
</ul>

<h3>项目专用空间</h3>
<p>我们围绕特定项目构建频道和团队，这也是我们大部分协作发生的地方。项目通常将大型计划所需的多个功能组合在一起，例如 <a href="https://zed.dev/channel/git-1-0-25944"><code>git 1.0</code></a>, <a href="https://zed.dev/channel/edit-predictions-v2-23075"><code>edit predictions v2</code></a>, <code>delta db</code>, 和 <code>cloud</code>。在这些频道中，项目成员作为主持人分享他们的 Zed 代码库实例，供团队协作。频道笔记通常包括项目成员名单、目标、我们旨在通过此项工作解决的 <a href="https://github.com/zed-industries/zed">GitHub Issues</a> / <a href="https://github.com/zed-industries/zed/discussions">Discussions</a> / 项目看板的链接，以及项目的总体进度。</p>

<h3>个人专注空间</h3>
<p>在我们的树状结构中，有一个 <code>people</code> 频道。我们鼓励员工在这里添加一个以自己名字命名的子频道。这些是我们的个人工作区——我们的“虚拟隔间”。当队友在一个个人频道中时，这往往发出一个信号：“我需要一些低头专注的时间来完成这项任务，但如果你有事，欢迎随时过来。”团队中的每个人使用这些频道的方式略有不同。我经常使用我的频道来组织我想写的博客文章的内容。</p>
<p><strong>趣闻</strong>：<em>这篇博客文章最初就是在我的 <code>blog</code> 子频道中列的大纲。</em></p>
<p>敏锐的观察者可能会注意到，在上图的这些频道旁边没有头像。这些频道空着并不罕见，因为团队通常更喜欢在可能的情况下进行协作！</p>
<hr>
<p>我们的虚拟办公室与任何其他实体办公室并没有太大不同——我们有指定的空间用于会议、项目工作和个人专注时间。我们构建的频道树结构旨在支持那些能让我们有效运营公司的工作流程，但你可以根据最适合你团队需求的结构来构建你们的频道树。</p>
<h2>我们的前进方向</h2>
<p>虽然 Zed 中的协作让我们能够在 Zed 内部运营 Zed Industries，但这仅仅触及了我们<a href="https://zed.dev/blog/sequoia-backs-zed">对团队工作愿景</a>的皮毛。我们正在构建一个未来：协作是持续的对话，而不是离散的提交——每一次讨论、编辑和见解都随着代码的演变而保持链接，对队友和 AI 智能体都触手可及。</p>
<p>走到这一步并非一帆风顺。多年来，我们曾暂停协作功能的开发，转而专注于用户频繁请求的功能——<a href="https://zed.dev/ai">AI 辅助工具</a>、<a href="https://zed.dev/debugger">调试</a>、<a href="https://zed.dev/windows">Windows 支持</a>和 <a href="https://zed.dev/git">git 支持</a>——但我们对 Zed 的首要目标<strong>从未改变</strong>。随着我们在基础必备功能上与其他编辑器达到同等水平，这些“绕路”变得越来越少，使我们能够重新专注于我们最兴奋的事情：构建最棒的多人软件开发工具。</p>
<p><em>目前的协作功能被视为 <code>alpha</code> 测试版，目前所有人均可免费使用！欢迎查阅 <a href="https://github.com/zed-industries/zed/tree/main/crates/collab">源代码</a>。</em></p>
