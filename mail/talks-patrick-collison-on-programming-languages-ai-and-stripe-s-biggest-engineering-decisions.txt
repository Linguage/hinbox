12:Tba95,<h1>访谈概述</h1>
<p>本次访谈由 AI 代码编辑器 Cursor 的 CEO Michael Truell 对话知名支付巨头 Stripe 的联合创始人兼 CEO Patrick Collison。在访谈中，Patrick 回顾了他早年使用 Smalltalk 和 LISP 语言的编程经历，探讨了为何现代编程环境仍需向早期“小众”语言借鉴集成开发环境（IDE）的理念。他分享了 Stripe 在早期技术选型（如 Ruby 和 MongoDB）上的“大爆炸”时刻，以及近期对 Stripe 核心 API 进行深度重构（V2版本）的经验与教训。</p>
<p>此外，两人深入探讨了 AI 的发展对宏观经济（如 GDP 与全要素生产率）的影响、“进步研究（Progress Studies）”在当下的重要性，以及利用 AI 破解复杂人类生物学疾病的巨大潜力（Patrick 参与创立的 ARC 研究所正致力于此）。最后，Patrick 作为深度用户，对 Cursor 提出了关于如何通过 AI 提高软件的“工艺与美感”、避免生成“垃圾代码”的独到建议。</p>

<h1>访谈译文</h1>
<h2>Patrick 的第一次创业与 Smalltalk</h2>
<p><strong>Michael Truell:</strong> 很高兴能邀请到您。谢谢，感谢您的到来。</p>
<p><strong>Patrick Collison:</strong> 谢谢你们邀请我，很高兴来到这里。</p>
<p><strong>Michael Truell:</strong> 我听说您的第一次创业项目是用 Smalltalk 语言写的，能解释一下吗？</p>
<p><strong>Patrick Collison:</strong> 我不知道这有什么好解释的，它就是最好的编程语言（笑）。在那之前，我一直在用 LISP 和 LISP 的方言工作，实际上我当时在开发基于 LISP 的 Web 框架。当我们准备建立第一家初创公司时，最初是用 Rails（Ruby on Rails）实现的。但与 LISP 相比，我觉得那种开发过程有些令人沮丧。我们不需要深入讨论全部细节，但我认为基于 Continuation（延续性）的 Web 框架才是实现 Web 应用程序的正确方式。</p>
<p>而 Ruby 里没有 Continuation 机制，也没有基于此的框架。在四处寻找之后，我发现当时正好有一个用 Smalltalk 编写的优秀框架，所以我决定试用一下。随后我发现，Smalltalk 实际上是一个极其有趣的开发环境，它具备了许多我非常欣赏的 LISP 的特性。比如，它是一个完全交互式的环境，带有一个真正的调试器，你可以在某个 Web 请求执行到一半时，或者深入到某个堆栈跟踪内部时直接编辑代码。</p>
<p>举个例子，你在处理某个 Web 请求时遇到了错误，你可以直接编辑代码来修复它，然后在堆栈的更高层级恢复执行，这样整个 Web 请求就会顺利完成。相比之下，传统的反馈循环非常烦人：你必须添加一些日志打印，然后像做二分查找一样去定位问题，最后可能还要花一个小时去部署一个修复版本。而在 Smalltalk 里，你只需检查堆栈帧，看看哪个变量的值不对，修复它，跳回上层，点击继续，一切就跑通了。</p>
<p>总之，为了寻找这个基于 Continuation 的 Web 框架，我意识到，与 Ruby 乃至基本上所有其他主流编程语言相比，Smalltalk 拥有一个强大得多的开发环境。所以我们决定在公司里使用它。事后看来，我不知道这算不算一个糟糕的决定。人们可能会觉得这很糟糕，因为很难招到人，很难扩大规模之类的。但实际上招人并不难，或者说，虽然没人懂这门语言，但教他们非常容易。</p>
<p><strong>Michael Truell:</strong> 他们在加入公司前懂这门语言吗？</p>
<p><strong>Patrick Collison:</strong> 不懂。但他们学得非常快，聪明人学编程语言总是很快的。所以我认为“非主流语言”本身不应该成为不使用它的理由。那家公司最后没有成功，我认为原因是另外的，仅仅是因为当时的商业点子不够强大。不过，我们后来在创立 Stripe 时也选择了 Ruby。所以，我想当初带来的收益可能并没有我期望的那么大。</p>
<p><strong>Michael Truell:</strong> 当那家初创公司被收购时，收购方也认同您对 Smalltalk 的热情吗？当时的情况是怎样的？是不是管理层在幸福的无知中，把这个 Smalltalk 代码库强加给了一群毫无防备的开发者，然后他们不得不硬着头皮接手？程序员和管理层之间的动态是怎样的？那个 Smalltalk 代码库后来怎么了，现在还在哪里运行吗？</p>
<p><strong>Patrick Collison:</strong> 我倒是希望它还在，但我有 99% 的把握它的答案是“不在了”。收购我们的那家公司主要看重的是人才收购。所以，代码库本身并没有那么重要。</p>
<p><strong>Michael Truell:</strong> 明白了，所以它几乎立刻就被弃用了。</p>
<h2>LISP 聊天机器人与神经网络的擦肩而过</h2>
<p><strong>Michael Truell:</strong> 我还听说，您最早的编程项目之一是用 LISP 写的一个 AI 聊天机器人。那似乎是一个 MSN 的客户端。</p>
<p><strong>Patrick Collison:</strong> 我不知道你从哪里扒出来的这些，但这确实是真的。</p>
<p><strong>Michael Truell:</strong> 我听说你当时被“让它通过图灵测试”这个想法给彻底迷住（Nerd sniped）了。</p>
<p><strong>Patrick Collison:</strong> 是的。</p>
<p><strong>Michael Truell:</strong> 我很好奇，您错过了什么？为什么您当时没有发明出 ChatGPT？说得稍微严肃一点，它是如何运作的？当时的神经网络处于什么状态？您当时考虑过使用任何我们今天使用的技术的雏形吗？</p>
<p><strong>Patrick Collison:</strong> 那个项目是一个使用 MSN Messenger（当时非常流行）的小程序。提到 MSN，这大概给我打上了一个非常精确的时代烙印，它属于即时通讯解决方案编年史中一个特定的“沉积层”。</p>
<p>它当时是一个非常简单的基于贝叶斯（Bayesian）的下一个词预测器，并没有什么真正复杂的东西。如果说有什么特别之处，大概就是它的训练数据来源于它自己在 MSN Messenger 里的对话记录，而不是通用的文本语料库。它运行得相当不错，后续较好的版本甚至能往后看几个词。</p>
<p>如果面对那些真正带有怀疑态度、试图去进行专业辨别的测试者，它显然从未真正通过图灵测试。但如果面对那些毫无防备的人，它确实通过了某种弱化版的图灵测试，人们最终会和它进行相当漫长的对话。</p>
<p>那是我发现 LISP 的契机之一，我记得 Peter Norvig 写的《人工智能编程范式（Paradigms of AI Programming）》对我影响深远，里面介绍了很多有趣的方法。但我几乎可以肯定，那本书里没有任何关于神经网络的内容。我也读过一些 Marvin Minsky 关于神经网络的文章（比如《心智社会》），但我从未真正认真研究过它们。</p>
<p>实际上，我对遗传算法进行了大量实验。在个人电脑上，它们似乎更实用。训练神经网络需要消耗大量的计算资源。所以我用遗传算法做了很多实验。顺便说一下，我现在使用的是 Dvorak 键盘布局，因为打字比 QWERTY 舒服得多。我弟弟 John（Stripe 联合创始人）也是，所以没人能用得惯我们的电脑。我当时写了一个遗传算法优化器，试图找出最优的键盘布局。结果证明，使用遗传方法得出的最优解基本就是 Dvorak 布局。我沿着那个兔子洞钻得很深，但从未真正涉足神经网络。我想，这大概就是为什么我没有发明出 ChatGPT 的原因之一吧——当然，肯定还有其他 70 多个原因。</p>
<h2>小众编程语言与 IDE 的启示</h2>
<p><strong>Michael Truell:</strong> 有一段您在卖掉 Automatic 公司后接受采访的旧视频，当时有人问您关于 Smalltalk 的问题（我也是从那里得知这个奇特事实的）。当时人们问您为什么选它，您提到的其中一点是，您喜欢 Smalltalk 和 LISP 风格语言的某些特性。您当时（大约是 2008 年）预测，主流的 C 风格编程语言将越来越多地借鉴这些古老语言的理念，而这在 JavaScript 和 Python 生态系统中似乎已经应验了。您认为在那些古老、小众的编程语言中，还有哪些被低估的想法应该被主流吸收？</p>
<p><strong>Patrick Collison:</strong> 很多想法被 JavaScript 生态系统借用，这种方式挺有趣的。它们以一种奇特的方式被吸收，比如通过 Web Inspector（网页检查器）。在那里，你拥有某种意义上最丰富的运行时（Runtime）环境，也是大家最常接触的。我不认为 JavaScript 拥有一等公民的堆栈帧（First class stack frames），也许有某种奇怪的扩展能做到，但 ECMAScript 本身没有。但拥有首等堆栈帧显然能让你做很多其他事情，也许这有点太具体了。</p>
<p>我认为，有一个基本的理念（也许这也是 Cursor 未来的方向）：将环境视作“开发环境（Development Environment）”而不仅仅是“文本编辑器（Text Editor）”，这才是正确的思路，这也是我希望回归的东西。这就是 LISP 机器和 Genera 系统拥有的东西，在某种程度上也是 Mathematica 拥有的东西，也是 Smalltalk 拥有的东西。</p>
<p>我们现在落得这样一个开发环境：运行时（Runtime）与文本编辑、以及代码运行的环境之间存在如此巨大的割裂，我认为这是一个巨大的错误。这三种环境本可以共存在同一个地方。直到今天，我仍然经常使用 Mathematica，不是因为我在做多么深奥的符号数学运算，而是因为它是一个极其高效的开发环境。</p>
<p>当然，有了大语言模型（LLM）后，这一点可能略有减弱，因为 Mathematica 不支持类似 Cursor 那样的提示词驱动开发。但我认为，那才是我希望其他人能借鉴的核心理念。VS Code 在某种程度上朝着这个方向迈出了一小步，但我们完全可以走得更远。</p>
<p>例如，我非常希望看到这样一种体验：当我将鼠标悬停在一行代码上时，我希望能看到该代码或函数的性能分析信息及其运行时特征；我希望能看到覆盖在上面的日志和错误信息；当我悬停在一个变量上时，我希望能看到它在生产环境中最常见的取值。我想要这些丰富而深度的集成体验。</p>
<p><strong>Michael Truell:</strong> 您是“基于原则发明（Inventing on Principle）”等理念的拥趸吗？</p>
<p><strong>Patrick Collison:</strong> 是的。我是 Bret Victor（“基于原则发明”的提出者）的超级粉丝，他太不可思议了。</p>
<p><strong>Michael Truell:</strong> 您去过 Dynamicland（Bret 创办的研究实验室）吗？</p>
<p><strong>Patrick Collison:</strong> 是的，去过，并且我也支持过它。所以我是 Bret 的铁杆粉丝。不过，我与他可能存在一点分歧，或者说让我产生共鸣较少的地方在于：Bret 非常热衷于为现象寻找图形化和视觉化的表示方法。我认为这在某些领域非常有效，比如他用来演示某些理念的动力系统。</p>
<p>但我认为，要为任意系统（比如 Stripe 的各个部分）找到这样有用、具有空间连续性的视觉表示通常是非常困难的。我不太清楚那会是什么样子，就算我们能找到，我也不确定它到底有多大用处。也许这只是我个人的偏好，我的推理方式更多是符号化和词汇化的，而不是视觉化和图形化的。他所致力于的那种颠覆范式的尝试，我认为是极具令人钦佩的。</p>
<h2>编程的未来演进与 AI 的角色</h2>
<p><strong>Michael Truell:</strong> Cursor 未来会打造一个真正的集成开发环境吗？我们目前正在测试一些想法，比如让 AI 越来越多地退到后台，让它自己运行代码并对输出做出反应。我们认为这些功能应该很好地协同工作。</p>
<p>我们在心流体验（Inflow）、速度和控制权上投入了大量精力，我们认为这对于 AI 来说极其重要——将控制权交还给程序员，让他们理解 AI 生成的所有内容；同时也要给他们极快的迭代循环，因为程序员最讨厌等待。但在某些情况下，我们认为现在可以让 AI 先去“思考”一会儿，然后再带着结果回来，让 API 变得更像是在与另一个人交流。</p>
<p>我们希望所有这些都能无缝协同：AI 可以带着 70% 的半成品回来，然后你可以非常快地将其带入前台，与之协作，然后再将任务甩回后台。作为让 AI 在后台花大量时间思考的一部分，为了让这种思考有用，它必须能够运行代码并做出反应，否则它就只是在盯着自己写的代码发呆。</p>
<p>也许现在应该是我回答问题而不是提问了。但是您认为 5 年后，我在 Cursor 里看到的主要东西还是代码吗？还是别的东西？</p>
<p><strong>Patrick Collison:</strong> 我认为可能会是别的东西。这当然是个巨大的简化，但当你定义一个软件时，大致分为两部分：一是逻辑组件，工程师花大量时间设计软件的具体运作方式；二是为了最终用户应用程序和带有 GUI 的软件，这里有一个视觉组件。</p>
<p>我认为未来的世界中（可能是你们实现的，也可能是别人实现的），你与 AI 交互的方式不再仅仅像是一个人类助手，或者是一个站在你身后预测你下一步动作的工具。相反，它会更像是一种高级的编译器或解释器技术。</p>
<p>这可能会引领我们进入这样一个世界：编程语言本身发生改变，它们可以变得不那么形式化，层次变得更高。编程将更多地关乎“你想要什么”，而不是“你如何实现它”。我认为它不一定会变成像 Google Doc 那样的形式，你会保留一些编程中的优秀概念，比如将某处的逻辑命名，然后在其他地方复用它。我认为这也将更多地融入软件视觉界面的设计中，世界可能开始向直接操作 UI 的方向发展。当然，这些都还是相当长远和实验性的想法。</p>
<p>总的来说，让我感到有趣的是，过去 20 年来，我们在编程范式上的实验并没有我们想象的那么多。我们今天讨论的很多东西其实都是 70 年代或 80 年代的概念。如今的开发者数量显然比过去任何时候都多，但在某种意义上，实验的窗口感觉并没有被真正打开。当然，JavaScript 生态系统和其他一些领域做了一些很酷的事情，Rust 和 Go 等在语言层面也有很多实验，但在开发环境的层面，不知为何，实验比预期的要少。也许是因为现在的事情变得太困难和复杂了吧。</p>
<p><strong>Michael Truell:</strong> 我同意。这也是我们正在努力的方向，希望这能在一定程度上解释 Cursor 的成功——你们是这么长时间以来第一批真正认真对待这个领域的人。</p>
<p>我们确实受益于“天时地利”，现在有了一套绝佳的“新颜料”供我们作画。同时，编程语言本身存在巨大的锁定效应：一方面是大脑神经元的锁定，编程语言是程序员用来定义计算机如何运作的复杂 UI，人们学习了这些语言后，不愿意再去学太多新东西；另一方面是已有代码库的锁定，你有一堆用某种语言编写的逻辑需要维护。</p>
<p>我们希望，随着 AI 编程变得越来越好，我们能解决大型专业应用开发中的痛点。当数百人共同处理数百万行逻辑代码时，代码库的沉重感会让你喘不过气来。那种身处全新代码库时的轻松感消失了，一切都变成了苦差事，牵一发而动全身，系统变成了难以理清的“大泥球”。减轻现有逻辑的重量，让修改代码变得毫不费力，我认为这是 AI 可以让编程变得更好的领域之一。</p>
<p><strong>Patrick Collison:</strong> 今天有人在 Twitter 上说（也许是 Andrej Karpathy，但也许我记错了，现在很多关于 AI 写代码的金句都被归功于他，就像丘吉尔或爱因斯坦一样）。那个人观察到：提示 AI 生成代码是一回事；而另一个 AI 可能大有作为的领域，是代码库的“美化”和重构。</p>
<p>你可以想象，你在前台快速产出了一堆略显笨拙、重构得不够完美的“代码残渣”；然后在夜里，这个 AI 会在后台跟进，帮你把代码梳理得漂漂亮亮。</p>
<p>我唯一上过的一门计算机科学课是 Gerry Sussman 教的。他把那门课称为“大规模符号系统”，但实际上他想传授的核心理念是：如何创建易于修改的代码库、环境和抽象。这门课里没有要求从头写代码的作业，所有作业都是关于修改现有系统的，要求你思考如何设计系统才能使这些（可能非常深度的）修改变得直接了当。</p>
<p>我认为这是一个美妙的想法。显然，在实践中，考虑到你要在今天或下周交付功能的紧迫压力，做到这一点通常非常困难。但如果你有一个 AI——很多时候在写代码时，你心里清楚“我本该用更优雅的方式来实现它，但我现在没有时间”——如果有一个 AI 能在背后帮我们清理和重构，那就太棒了。</p>
<h2>组织工程与长远的技术决策</h2>
<p><strong>Michael Truell:</strong> 很多开发者进入这个行业是因为他们喜欢创造，喜欢让屏幕上发生奇妙的事情，于是他们开始写代码。但随着发展，很大一部分开发者最终会发现，他们想要创造的软件太庞大，无法独自完成，必须依靠团队合作。于是他们可能成为工程经理、总监，或者自己创业。这时，大部分工作就不再是敲代码，而是变成了协调人员。您认为，编程中的哪些理念有助于对组织进行“编程”，从而让一群人共同构建好软件？</p>
<p><strong>Patrick Collison:</strong> 很有意思。我认为是极其重视 API 和数据模型。</p>
<p>如果要让我在 Stripe 重新来过，虽然有无数细节我会做出不同的选择，甚至一些重大的改变，但我能预见并确信带来好处的一点是：在 API 和数据模型上投入比当初更多的时间。</p>
<p>部分原因在于康威定律（Conway's Law）的效应。这两个因素最终会塑造你的组织结构。如果你没有深刻内化这一点，你可能无法像你希望的那样控制组织的动态。弱化版的康威定律认为它塑造了你的组织；我认为强化版则是，它深刻塑造了你的战略和业务成果。</p>
<p>举个不是完全对等的例子，我经常思考为什么 iOS 的软件生态系统在很长一段时间内（甚至可能直到今天）都比 Android 的生态系统更具活力、生命力和成功。这两个生态系统之间有很多不同之处，现在 Android 设备的数量显然远超 iOS 设备。</p>
<p>但我认为，应用开发者倾向于首选 iOS、先在 iOS 上发布，甚至 iOS 版本做得更好，很大程度上是因为 iOS 早期的框架和抽象设计就比 Android 更优秀。这就是一个正确的 API 设计、正确的抽象设计最终产生重大商业影响的案例。</p>
<p>现在有一种感觉，人们认为在技术领域一切变化得太快，不需要在这些设计上深思熟虑，因为你做出的任何假设可能两年后就过时了。但在实践中，我认为这并不成立。正确的 API 设计、适当的抽象和数据模型是可以经久不衰的。</p>
<p>比如第一代 iOS 系统，开发者使用的许多类名都带有 <code>NS</code> 前缀，这代表的是 NextSTEP（乔布斯离开苹果时创办的公司）。这是一个 API 设计存活了二十多年甚至更久的例子。就 Stripe 而言，公司现在已经 15 岁了。有很多我们 15 年前设计的东西今天仍在使用。这可以说是好坏参半，好在它们经受住了时间的考验，坏在我们在某种程度上仍要忍受它们早期的设计缺陷。</p>
<p><strong>Michael Truell:</strong> 确实如此。关于这一点，我曾与硅谷一家非常成功的大型私企的工程负责人交流过。他们公司的代码库主要是用 Scala 写的。他们说，初创公司的早期阶段就像是一场“大爆炸（Big Bang）”。当时那些疲惫不堪、劳累过度、也许咖啡因摄入过量的初创团队成员，随性地做出了一些初步的技术决策，而这些决策最终决定了未来数百名专业工程师的工作生活。选择 Scala 就是其中之一，他们现在不得不在其缺陷中共存。</p>
<p>那么，Stripe 的“大爆炸”留下了哪些产生深远影响（无论好坏）的初始条件，是你们至今仍在经历的？</p>
<p><strong>Patrick Collison:</strong> 我认为这个隐喻听起来很真实。也许这里存在一点“幸存者偏差”，比如，那些我们早期做出但后来从未改变的决策，才是我们现在继续共存的决策，这有点同义反复。当然，我们早期做出的一些设计决策，今天显然已经不再适用。比如早期版本的 Dashboard（仪表盘）与今天的设计大相径庭。</p>
<p>但反过来也成立。我们在早期决定使用 MongoDB 和 Ruby。它们至今仍是 Stripe 非常基础的技术。为了让 MongoDB 达到我们所需的容错性、分布式能力、持久性和可靠性，我们不得不自行构建了大量的基础设施。去年，Stripe 关键 API 的可用性达到了 99.99986%，这意味着全年的不可用时间只有 44 秒。我们认为其他公司可能不会公布这么精细的统计数据，但这应该是行业最佳水平。</p>
<p>所以，我们的存储团队及其他许多团队构建的一切最终成功运作了，但那确实是一个极其关键的初始决策。至于 Ruby 也一样，虽然公司有时会在发展过程中更换语言，但我感觉最初选择的语言往往会产生深远的影响。</p>
<p><strong>Michael Truell:</strong> 我听说在 Stripe 内部曾有过关于语言迁移的讨论。你们的一位联合创始人在早期（或者是你们个人早期经历中）曾提到过，有过堆积如山的关于是否迁移到 Java 的文档。</p>
<p><strong>Patrick Collison:</strong> 是的，这部分发生了。我们将一些关键服务用 Java 进行了重写。对于需要极高吞吐量的服务来说，如果你足够“折磨” Ruby，可能再用 C 语言重写一些热点路径，你确实能让它跑得相当快，但你往往是在与内存分配器抗争，甚至连 Ruby 的字符串处理效率都不够高。所以我们将某些特定服务用 Java 重写了。现在我们两者都在使用。</p>
<p><strong>Michael Truell:</strong> 早期为什么选 Ruby，或者有没有考虑过其他语言？当时的 RFC（技术征求意见）和决策过程是怎样的？</p>
<p><strong>Patrick Collison:</strong> 所谓的决策过程其实就只有我和 John（笑）。我们坐在沙发上说：“我们要用 Ruby 吗？”“好吧，就用它。”</p>
<p><strong>Michael Truell:</strong> 是某篇博客文章打动了你们吗？还是当时它在开源社区的声誉？</p>
<p><strong>Patrick Collison:</strong> 在做上一家公司时，我曾亲自编写了一个基于对象的数据存储系统，因为我不太喜欢 SQL。我认为在应用程序的业务领域与 SQL 原生表达的领域之间，存在太大的转换偏差。</p>
<p>使用 SQL，你显然必须将数据压缩成相对受限的原始形式。而在应用程序中（以 Stripe 为例），你可能有一个“金钱”的概念，它并不能与你碰巧使用的特定 SQL 数据库表示金钱的方式完全吻合。</p>
<p>出于某种原则上的坚持，我当时非常抵触 SQL。我并不是在肯定这种做法，但这恰恰说明，就像这次采访所展示的，我满脑子都是关于技术的奇思怪想。而在做 Stripe 时，我们希望能比上一家公司更主流一些，技术选择上不那么异端。所以我们没有用 Smalltalk，而是选择了 Ruby（相比之下显得更主流）；同样，我们没有自己写对象数据库，而是走向相对主流的 MongoDB，它作为一种对象数据存储，依然赋予了我们很大的灵活性。</p>
<p>这算是挺好的决定。但我刚才说的这些可能足以取消我未来为任何其他公司做技术决策的资格（笑）。</p>
<h2>Stripe 的核心 API 重写计划</h2>
<p><strong>Michael Truell:</strong> 对于“Stripe 第二版（Stripe 2）”，您会有什么不同的做法吗？</p>
<p><strong>Patrick Collison:</strong> 我们在公开场合还没怎么讨论过这个话题。答案可能有点像周恩来总理评价法国大革命的那句名言：“现在下结论还为时过早”。</p>
<p>大约在 2022 年，回到我们刚才关于数据模型和抽象的讨论，我们意识到 Stripe 中的几个核心抽象并不是正确的长远方案，我们必须修复它。因此，我们设计了一系列 V2 版本的 API。幸运的是，我们在 Stripe 早期就预见到了这种可能性。大家熟悉的 Stripe 的 REST URI 大多数都带有 <code>/v1</code> 前缀，这个前缀从 2010 年起就存在了。到了 2022 年，我们决定是时候升级命名空间了。</p>
<p>我们设计了这些新 API，并在今年开始推出。</p>
<p><strong>Michael Truell:</strong> 恭喜。</p>
<p><strong>Patrick Collison:</strong> 谢谢。我们对它将实现的新功能感到非常兴奋。不涉及太多晦涩的技术细节，简单来说，它将解决这样的问题：过去我们将“最终客户（End customers）”、“子账户（Sub-accounts）”以及不同类型支付的“收款人（Recipients）”作为独立实体分开表示。现在，我们将它们统一到同一个实体表示中。</p>
<p>在某种层面上，这显然是正确的答案，它已经开始改变我们一些客户的业务方式。因为他们可以让自己的用户执行各种操作而无需重复输入信息，或者将同一个账户跨国使用等等。</p>
<p>这可以说是一段漫长的旅程。之所以漫长，是因为仅仅孤立地定义这些新 API 并没有多大用处。光是定义它们其实很容易，困难在于如何让它们与 Stripe 现有的所有系统互操作，并构建翻译层。更困难的是与我们的客户一起找出一条合理的升级路径，因为我们控制的是自己的代码库，而不是他们的。</p>
<p>虽然不想夸大其词，但在某些方面，这感觉有点像芯片架构的指令集迁移（Instruction set migration）。指令集本身并不难，但所有共存与兼容的问题才是真正的难题。不管怎样，它今年终于要发布了，我们对此非常兴奋。</p>
<p><strong>Michael Truell:</strong> 您从中学到了什么教训？对于那些涉及重写，或者需要考虑几十年跨度抽象架构的项目，有什么更宏大的启示吗？</p>
<p><strong>Patrick Collison:</strong> 我比较直白的回答是：尽可能把所有看似合理能够统一的东西统一起来。</p>
<p><strong>Michael Truell:</strong> 你们是如何测试 V2 版本的初步设计想法的？</p>
<p><strong>Patrick Collison:</strong> 主要是靠设计它们的人。对了，我再补充一个教训。这可能听起来有点廉价。</p>
<p><strong>Michael Truell:</strong> 有没有一个“首席 API 设计师”作为幕后主脑统筹一切？是不是由一个人拍板，而不是某个工作组？</p>
<p><strong>Patrick Collison:</strong> 我们有工作组，但也确实需要有一个人，他比任何人都更理解并对整体架构负责，我认为这是必不可少的。</p>
<p>关于另一个教训，我的建议是：让任何有可能成为“多对多（N by M）”关系的设计直接支持多对多。如果你只支持一对多（1 to N）或多对一（N to 1），哪怕现在看起来绝对不可能出现多对多的情况，不可避免地，未来某天你终究会需要它。你可能认为一家公司绝不可能被另外两家公司共同拥有，但事实证明，现实空间中的每一种排列组合，最终都会发生。</p>
<p>至于我们如何知道这些是正确的 API？一方面是通过向客户展示早期版本获取反馈；另一方面，设计这些 API 的人都在旧版本的缺陷中浸淫和痛苦了许多年。所以我们确实带着强烈的个人见解。</p>
<p>但即便是强烈的见解，有时也可能预测错误、过度外推或过度设计。所以客户验证和反馈的循环极其重要。同时，我们要求自己亲自编写新世界中的集成代码，这非常重要。就像 Java 修复了 C/C++ 内存管理的问题，但代价是极度的冗长和繁琐的系统开销。为了防止我们无意中过度设计，我们强迫自己编写大量模拟真实业务模型的 API 调用代码，以确保它看起来是对味的。</p>
<p>当然，我现在还不想过早地宣扬我们的做法有多成功。我非常乐观，但目前进度大概在 60% 到 70%，并没有完全结束，所以我还不能过早宣布胜利。</p>
<h2>Patrick Collison 个人如何使用 AI</h2>
<p><strong>Michael Truell:</strong> 您本人是如何使用 AI 的？</p>
<p><strong>Patrick Collison:</strong> 最主要的方式比较常规，就是大量使用各种 LLM 聊天工具。</p>
<p>我主要用它们来回答我感到好奇的事实性或经验性问题。对于深度研究类问题，我并不总是使用“深度研究（Deep Research）”模式，现在模型使用工具和自主浏览网页的能力越来越强，对深度研究模式的依赖也相对减少了。</p>
<p>我其实很希望它们能用于写作，但我总是对它们写出来的东西感到不满。所以我很少用它们来写文章，甚至连修改润色我自己的文章也很少用。</p>
<p><strong>Michael Truell:</strong> 随着模型的进化，您在写作方面看到什么改进了吗？我也有同感，它们写的东西有一种令人惊讶的“同质化”和“套路感”。即使我提示它不要太套路，或者插入特定人物的名字，效果依然很差。</p>
<p><strong>Patrick Collison:</strong> 是的。有人告诉我基础模型在这方面做得更好，而基于人类反馈的强化学习（RLHF）所带来的“规范化（Normification）”，使得模型的输出掉入了一个特定的“吸引子盆地（Attractor basin，指生成风格高度趋同）”。</p>
<p>我确实没能有效地将它们用于写作。有人说 Claude 或 OpenAI 的新模型比早期模型好，相对而言可能是真的。我不想显得自吹自擂，好像我是一个多么有天赋的作家。我并不觉得自己是，只是我的个人风格与模型的“个人风格”不同。在某种自我中心的意义上，当我写作时，我就是想用我自己的风格。</p>
<p>所以我经常用它们来查阅事实性内容，觉得非常棒。即使在看书时，我最近一直在用 Grok 的语音模式，我在阅读时会顺便问问题，Grok 就在后台倾听并给出解答，这非常有帮助。当然，我也用 LLM 写代码，这通常是通过 Cursor 来完成的。</p>
<h2>AI 对宏观经济的影响：GDP 与 TFP</h2>
<p><strong>Michael Truell:</strong> 我们采访您，因为您堪称当今“软件实业家”的典型代表：您经营着一家成功的大型软件公司，您从程序员起步转型为管理者，而且您的公司还致力于为开发者提供工具，您处于多个领域的交集。同时，我们也是在采访另一面的 Patrick Collison：一位业余经济学家和世界观察者。</p>
<p>既然 AI 已经到来，“进步研究（Progress Studies）”这门学科是不是要终结了？还需要它吗？</p>
<p><strong>Patrick Collison:</strong> 恰恰相反，我认为对“进步研究”的需求增加了（我不是指某个带首字母大写的特定专有名词，而是指这类思维方式）。这种研究所试图回答的问题，现在变得更加紧迫。</p>
<p>我认为未来的自由度正在增加。有一种盲目乐观（Panglossian）的观点认为，AI 会神奇地解决所有问题。预测未来总是很难的，但我不相信这种观点，并且从迄今为止的证据来看，历史轨迹也并非如此。因此，我们如何使用这些技术、做出什么样的决定、追求什么样的人类福祉，这些判断将变得至关重要。</p>
<p>也许五年前你能对“进步研究”提出这样的批评：这些问题都很好，但世界似乎正走在一部自动扶梯上，注定要到达某种特定的结局。但今天的世界已经让人感觉不到这种必然性了，或者说至少比以前弱了很多。</p>
<p><strong>Michael Truell:</strong> 是因为全球地缘政治，还是其他原因？</p>
<p><strong>Patrick Collison:</strong> 主要有三个方面：一是全球事务的整体局势。二是抱负和理想正受到越来越激烈的争议，如今在美国，甚至很难界定左派和右派到底代表什么，我们现在看到一个政党支持关税，另一个反对关税，但这与历史上的倾向完全反过来了。三是技术因素，首当其冲的是 AI，但在我们这个行业还有稳定币，以及中国在无人机、机器人、电池和太阳能等未来技术领域作为绝对制造强国的崛起。</p>
<p>所以，在很多方面，我感觉未来的图景变化极大。Peter Schwartz 有一个概念叫“Schwartz 窗口”，指的是在未来若干年内“可预见的未来”的时间跨度。如果在 2005 年展望 2015 年，这个窗口是相当狭窄的，而 2015 年的世界也确实与我们在 2005 年预期的差不多。但今天在 2025 年展望 2035 年，这个窗口感觉极其宽广。所以我认为这些进步研究的问题变得更加迫切了。</p>
<p><strong>Michael Truell:</strong> 您曾公开表示，人们应该更多关注这个问题：为什么随着信息技术的发展，以及越来越多的资金和人才投入到科技领域，我们在生产率数据（TFP）上却没有看到相应的提升。那么现在数据看起来怎么样？我们能在经济数据中看到 AI 的影响了吗？</p>
<p><strong>Patrick Collison:</strong> 最近几天刚刚发表了一篇关于这个主题的新论文。我还没来得及细读，只是今天刚把它放进了待读列表，目前只看了摘要。它的主张是：人们实际上并没有观察到大语言模型的使用带来了生产率的提升。</p>
<p>我现在还无法评判他们研究的方法论是否严谨，也许深入了解后我会印象深刻并认为它极具可信度，也可能会觉得不屑一顾。但这只是我今天恰好看到的一个发现。</p>
<p>总体来看，过去两年美国整体的 GDP 增长情况确实好于我们的预期。在这个充满波动的时期，我们显然没有看到任何呈指数级起飞的证据。退一步说，如果我们将过去两年美国令人鼓舞的 GDP 数据归功于这些新技术，那么你也应该在其他国家看到同样的增长，因为这些技术（LLM）具有准公共产品的性质，所有人都可以使用。然而，美国以外的 GDP 增长并没有那么令人振奋。</p>
<p>我们并没有生活在一个全球经济由于 AI 而大规模加速的时期。现在一切还处于早期阶段，这些技术在经济中的扩散需要时间，且涉及极大的复杂性。</p>
<p>关于这一点，最后我想提一下 Jack Clark（Anthropic 的联合创始人之一）在与 Tyler Cowen 对话时所说的话。Anthropic 始终非常严肃地对待通用人工智能（AGI）乃至超级人工智能（ASI）的概念，Dario Amodei 也经常公开讨论此事。但 Jack Clark 预测，AI 将使每年的 GDP 增长率提升 0.5%。</p>
<p>我认为 Jack 是一个乐观主义者，而且如果考虑复利，每年 0.5% 的额外 GDP 增长实际上是非常庞大的。我不是说这个数字小，但考虑到这出自 Anthropic 联合创始人之口，我觉得这是一个非常有趣且脚踏实地的预测。</p>
<p><strong>Michael Truell:</strong> 您认为以 AI 目前在经济体中的形态，如果我们直接把趋势线拉长，我们需要制定不同于现有的经济生产率衡量标准吗？假设实际生产率提升了，AI 也在变得更好并如期部署，它是否应该自然反映在现有的经济数据中？</p>
<p><strong>Patrick Collison:</strong> 不，我不觉得需要新的指标。我不是说 GDP 是完美的，它当然有改进的空间，但在任何一个我们普遍认知的“经济”获得大规模提升的世界里，它必定会在 GDP 数据中体现出来。</p>
<h2>对人类生物学进行“编程”</h2>
<p><strong>Michael Truell:</strong> 我们什么时候能够对人类生物学进行编程？</p>
<p><strong>Patrick Collison:</strong> 我对此感到非常兴奋。在我们参与创立的生物医学研究机构 ARC 研究所（ARC Institute）里，我们正在致力于使用 DNA 等数据训练生物学基础模型。我们正在研究“虚拟细胞”。</p>
<p>总体而言，我们在尝试解决一个我直到深入生物学领域后才意识到的问题：人类其实从未真正治愈过任何一种“复杂疾病”。</p>
<p>可以将疾病大致分为几类：第一类是流感、感冒、新冠、结核病等传染性疾病。第二类是单基因疾病，比如亨廷顿舞蹈症，只有一种基因突变导致了疾病。第三类就是“复杂疾病”。在西方世界，复杂疾病基本上是我们在治愈了大多数致命传染病后遗留下来的难题。</p>
<p>绝大多数心血管疾病、癌症、自身免疫性疾病、神经退行性疾病等，都属于复杂疾病。对于其中某些疾病，我们可能有一些有帮助的疗法（比如治疗心血管疾病的他汀类药物），但对于它们中的任何一种，我们都不能说已经彻底“治愈”了，不能说我们已经在实质性细节上掌握了致病的因果路径。</p>
<p>我们的假设（这当然可能是错的）是，部分原因在于我们缺乏足以胜任这项任务的实验技术和“认知工具”。基因的“多效性（Pleiotropy）”——即它们对身体的各个部位、系统以及细胞内部机制产生的影响——具有惊人的组合复杂性。加上外部环境又是一个极其庞大且难以量化的因素，弄清这些疾病的病理和动态机制实在是太难了。</p>
<p>但在过去的 10 年里（其实稍长一些，但主要发展在过去 10 年），我们在生物学领域获得了三类新技术。</p>
<p>在“读取（Read）”层面，我们获得了极其强大的测序技术，特别是单细胞测序（如单细胞 RNA 测序），这使我们能在细胞级别进行观察。
在“思考（Think）”层面，我们迎来了神经网络、深度学习、Transformer 等技术的爆发式进步。
在“写入（Write）”层面，功能基因组学、CRISPR 技术以及“桥接编辑（Bridge editing，ARC 的一项技术）”等取得了巨大进展，这使得我们能够在细胞内进行极其精确、定向的干预。</p>
<p>如果你把这三者结合起来，我们现在在个体细胞层面上，具备了“读取”、“思考”和“写入”的能力。</p>
<p>这开始感觉像是一个全新的“图灵循环”，具备了它自身的完备性。我们要看看这种组合拳到底能在对抗复杂疾病方面发挥多大作用，这种系统性的方法是否足以揭示它们运作的深层规律。但我们充满了希望与期待。</p>
<h2>AI 时代意想不到的受益者</h2>
<p><strong>Michael Truell:</strong> 如果我们 Cursor 及业内其他公司成功实现了对当今编程的自动化，并将其替换为一种更高层次、更具生产力的软件构建方式（人们只需专注于定义他们想要软件变成什么样），您认为谁会成为最大的赢家？</p>
<p>有人说设计师会迎来复兴；有人说这会让那些极其聪明但不太擅长在电脑上写代码的研究生受益。在这样一个人们创造数字产品的门槛大大降低，且现有程序员的生产力出现飞跃式提升的世界里，您认为谁会是最意想不到的受益者？</p>
<p><strong>Patrick Collison:</strong> 我没有一个非常笃定的答案。显然有一些非常老套的经典答案，比如稀缺的实物资产。比如，也许我们应该做多旧金山的房地产，因为它是世界上最美丽的城市之一，且将永远如此。也许我们应该做多这些 AI 系统的输入原料，因为对它们的需求将呈抛物线增长，比如做多铜矿。也许我们应该做多地位商品（Positional goods）、名人效应或者泰勒·斯威夫特的音乐版权。</p>
<p>这些理论都很有吸引力。但我认为，在这个经济节点上最有趣的一点是其不可预测性、偶然性，以及结果对技术发展轨迹中具体假设的高度敏感。AI 在 5 年或 10 年后呈现出的具体形态，将在很大程度上决定这个问题的答案。当我回看过去几年时，我惊讶地发现，即使是那些表面上消息极其灵通的人做出的预测，也有那么多被无情推翻了。</p>
<p>我向很多人问过这个问题，但至今还没有听到任何一个让我深信不疑的答案。</p>
<h2>Stripe 对 Cursor 的期待</h2>
<p><strong>Michael Truell:</strong> 我们非常荣幸能够为 Stripe 提供服务并助力你们的使命。您希望我们构建什么新功能？我们该如何为 Patrick Collison 个人，或是为 Stripe 公司提供更好的 Cursor 体验？</p>
<p><strong>Patrick Collison:</strong> 你们实际上已经在让 Stripe 变得更好了。从我们的角度来看，继续保持你们现在的路线就是一个非常好的结果。</p>
<p>在 Stripe，目前有数百名、很快就会有数千名对 Cursor 极其热情的员工，他们每天都在使用 Cursor，并反馈说这极大地提高了他们的生产力。</p>
<p><strong>Michael Truell:</strong> 等着看经济数据体现出来吧（笑）。</p>
<p><strong>Patrick Collison:</strong> 经济体量太大了，这种扩散需要时间。</p>
<p>如果你们已经让这个过程变得更高效了，再要求更多似乎显得有些贪婪。要知道，Stripe 在研发和软件创建上的投入，超过了我们在其他任何单一项目上的花费。</p>
<p>但如果允许我自私一点，我有三个建议：</p>
<p>首先，是我们刚才讨论过的<strong>运行时特性与上下文的深度集成</strong>，我认为这非常有价值。
其次，是代码的<strong>重构与美化功能</strong>。如果你们能降低未来修改 Stripe 代码的成本，并提高系统架构的质量，那将彻底改变我们行动的自由度。
最后，在 Stripe，我们非常在乎被我们称为**“工艺与美感（Craft and beauty）”**的东西。</p>
<p>我们希望我们的软件设计精良、使用愉悦。这种“愉悦”不仅是指表面像素级别的 UI 漂亮，更深层次是指“它运转得极其完美”的那种愉悦——一种你可以配置好后就放手不管并完全信任它的感觉。</p>
<p>显然，人们对 AI 有一种担忧：它会导致大量“垃圾代码（Slop）”或粗制滥造的内容被生产出来，而不是创造出更多最顶尖、最优秀的作品。</p>
<p>我不知道 Cursor 具体应该做些什么，来确保这个世界创造出的是“更多顶尖的软件”，而不仅仅是“更多的软件”。但我认为这是一个非常有趣且至关重要的维度。</p>
<p>所以，除了显而易见该做的事情之外，这就是我的三点建议。</p>
<p><strong>Michael Truell:</strong> 太棒了。谢谢您，Patrick。</p>
<p><strong>Patrick Collison:</strong> 好的，谢谢你们的邀请。</p>
